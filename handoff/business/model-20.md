## 핵심 결론

소형 AI SaaS·GPT·웹앱은 기능을 많이 넣는 사업이 아닙니다. 성패는 **반복 사용되는 좁은 문제 × 첫 가치 도달 속도 × 사용자당 결제액과 AI·서버 비용의 차이 × 월 해지·결제 실패 × 장애·지원·보안 운영비**에서 갈립니다.

`문제 빈도 × 첫 가치 도달 × 활성 사용자 × 유료 전환 × 잔존 × 결제액 - AI·서버·PG·지원·환불 비용`

데모가 잘 작동해도 사용자가 매주 돌아오지 않거나 AI 호출비와 고객지원비가 결제액보다 크면 사업이 아닙니다. 첫 기능보다 반복 사용과 비용 상한을 먼저 설계해야 합니다.

## 성패를 가르는 것

### 문제는 좁고 반복되어야 합니다
- 관찰: “무엇이든 물어보는 AI”는 기존 도구와 차이가 약하고 결제 이유가 불분명합니다.
- 숨은 변수: 월 사용 빈도, 오류 비용, 기존 대안, 입력자료, 결과를 쓰는 다음 단계입니다.
- 작동 메커니즘: 한 직무의 반복 문제를 짧은 입력과 구조화된 결과로 해결하면 재방문이 생깁니다.
- 실행 판단: `누가`, `어떤 자료를 넣고`, `어떤 결과를`, `몇 번 쓰는가`를 한 문장으로 정합니다.
- 실패·전환 신호: 첫 주 이후 사용자가 돌아오지 않으면 기능 추가보다 문제 선택을 바꿉니다.

### 가입보다 첫 가치 도달이 중요합니다
- 관찰: 회원가입과 무료체험은 사용자가 결과를 얻었다는 뜻이 아닙니다.
- 숨은 변수: 가입 단계, 초기 데이터 입력, 첫 결과시간, 예시 데이터, 오류 복구입니다.
- 작동 메커니즘: 가입 직후 실제 업무 결과를 만들면 유료 전환과 재사용 가능성이 높아집니다.
- 실행 판단: 첫 화면에서 예시 입력과 결과를 보여주고 10분 안에 첫 결과가 나오게 설계합니다.
- 실패·전환 신호: 가입은 늘지만 첫 결과 완료율이 낮으면 온보딩과 입력 구조를 줄입니다.

### 사용자당 비용 상한이 있어야 합니다
- 관찰: 긴 문서·이미지·음성·반복 대화는 AI 호출비와 저장비를 빠르게 늘릴 수 있습니다.
- 숨은 변수: 입력 길이, 모델 선택, 재시도, 캐시, 파일 저장, 무료 사용량, 악용입니다.
- 작동 메커니즘: 무료 사용량과 고비용 요청을 통제하지 않으면 사용자가 늘수록 손실이 커질 수 있습니다.
- 실행 판단: 기능별 평균 비용과 최대 비용을 측정하고 저비용 경로·캐시·사용량 제한·추가 결제를 설계합니다.
- 실패·전환 신호: 활성 사용자가 늘수록 사용자당 손실이 커지면 가격·모델·기능을 즉시 바꿉니다.

### 유료 전환보다 월 해지와 결제 실패를 봅니다
- 관찰: 첫 달 할인과 무료체험은 결제를 만들지만 반복 가치가 없으면 다음 달 해지됩니다.
- 숨은 변수: 첫 결제 이유, 월 사용 빈도, 결제 실패, 환불, 해지 이유, 연간 결제 할인입니다.
- 작동 메커니즘: 사용자가 반복 업무 결과를 저장·재사용할 때 잔존이 높아집니다.
- 실행 판단: 월 신규·활성·유료·해지·결제 실패·재활성 사용자를 구분합니다.
- 실패·전환 신호: 신규 결제가 늘어도 월 해지와 결제 실패가 같은 속도로 늘면 광고를 중단합니다.

### 고객지원과 장애가 숨은 제품 기능입니다
- 관찰: AI 결과 오류, 로그인, 결제, 파일 업로드, 브라우저 차이로 문의가 발생합니다.
- 숨은 변수: 문의 유형, 응답시간, 장애 복구, 로그, 상태 공지, 환불 기준입니다.
- 작동 메커니즘: 로그·오류 메시지·셀프 도움말이 없으면 모든 문제가 운영자에게 집중됩니다.
- 실행 판단: 오류 코드, 사용 로그, 상태 안내, FAQ, 환불·장애 보상 기준을 준비합니다.
- 실패·전환 신호: 사용자당 지원시간이 줄지 않거나 장애가 반복되면 기능 추가를 중단합니다.

### 개인정보·권한·AI 책임을 분리합니다
- 관찰: 사용자가 계약서·상담·고객·사내 자료를 입력할 수 있습니다.
- 숨은 변수: 수집 목적, 보관기간, 외부 API 전송, 삭제, 관리자 접근, 모델 학습 사용 여부입니다.
- 작동 메커니즘: 데이터 흐름을 설명하지 못하면 신뢰·계약·법적 위험이 생깁니다.
- 실행 판단: 최소 수집, 비밀정보 경고, 역할별 권한, 삭제·백업·사고 대응 절차를 문서화합니다.
- 실패·전환 신호: 민감정보를 안전하게 처리할 근거가 없으면 해당 입력 기능을 막습니다.

## 핵심 현실 인사이트
- 구체적 관찰: 가입·무료체험·첫 결제·활성 사용·잔존·순이익은 서로 다른 사건입니다.
- 남들이 놓치는 숨은 변수: 첫 결과시간, 사용자당 최대 AI 비용, 결제 실패, 지원시간, 데이터 삭제입니다.
- 결과가 달라지는 작동 메커니즘: 반복 문제를 좁히고 사용량·비용·오류를 통제하면 사용자가 늘어도 마진이 남습니다.
- 임계점·시간차·교환비용: `콘텐츠·광고 유입 → 가입 → 첫 결과 → 유료 결제 → 반복 사용 → 해지·환불·결제 실패 → PG 정산 → 원화 계좌 입금`
- 초보자가 먼저 측정하거나 결정할 것: 문제 빈도, 첫 결과, 사용자당 비용 상한, 월 가격, 지원 범위입니다.
- 계속할 조건: 활성·잔존이 개선되고 결제액에서 AI·서버·PG·지원비를 뺀 이익이 남는 경우입니다.
- 중단·전환 신호: 가입만 늘고 첫 결과·재사용이 없거나 사용자가 늘수록 손실이 커지는 경우입니다.
- 대표 사례·수치·반례: 월 1회 쓰는 범용 챗봇보다 매주 반복되는 견적·보고·분류 도구가 더 높은 결제 가치를 만들 수 있습니다.
- 반복 확인된 구조인지 단일 사례의 편집 판단인지: 활성·잔존·사용자당 비용을 우선하는 것은 사업성 판단이며 PG·API·개인정보 조건은 공식 확인이 필요합니다.

## 첫 수익과 실제 입금의 시간 차이

`네이버·YouTube·Instagram·Threads·당근·크몽·위시켓 유입 → 가입·첫 결과 → 신용카드·간편결제 → 무료체험 종료·월 결제 → 환불·결제 실패·해지 → PG수수료·AI·서버비 반영 → 원화 계좌 입금`

## 시장 현실

- 구매자: 소상공인, 1인 사업자, 프리랜서, 교육·상담·쇼핑몰·기관 실무자
- 유입: 네이버 검색·블로그, YouTube, Instagram, Threads, 당근, 크몽, 위시켓, 기존 고객
- 판매 방식: 월 구독, 사용량 결제, 건별 결제, 팀 요금, 구축·세팅 결합
- 결제수단: 토스페이먼츠·KG이니시스·나이스페이 등을 통한 신용카드·간편결제·계좌 관련 수단
- 수익 지급자: PG사 또는 판매 플랫폼
- 비용: AI API·서버·데이터베이스·파일 저장·모니터링·PG·환불·광고·고객지원·개발·보안
- 반복 가능성: 사용자당 마진과 잔존이 안정되고 지원시간이 통제되어야 합니다.

## 판매·수익·광고 경로

- 문제 해결 콘텐츠·무료 진단·당근 지역 노출 → 체험 → 첫 결과 → 유료 결제 → 반복 사용
- 네이버 검색광고·Meta 광고는 유료 고객 획득비용이 예상 고객 순이익보다 낮을 때만 시험합니다.
- 당근은 지역 소상공인의 소형 도구·세팅·방문 설명 유입 경로로 검토하고 외부 링크·서비스 홍보 정책을 확인합니다.
- 크몽·위시켓은 초기 구축·세팅 고객을 확보해 반복 문제를 찾는 경로로 활용할 수 있습니다.

## 공식 확인 사실

사업성 분석 단계에서는 토스페이먼츠·KG이니시스·나이스페이의 가입·정기결제·수수료·정산, 크몽·위시켓의 판매 조건, 당근 홍보 정책, AI API·서버·데이터 보관·개인정보·전자상거래 조건을 확정하지 않습니다. 다음 단계가 공식 문서로 확인해야 합니다.

## 현실 기준 교정

- 개인과 개인사업자의 PG 심사·통신판매업 신고·세금 처리를 확인합니다.
