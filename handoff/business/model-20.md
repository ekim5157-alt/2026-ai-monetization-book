# HANDOFF

## 기본 정보
- 모델 번호: 20
- 모델명: 소형 AI SaaS·GPT·웹앱
- 담당 역할: 사업성 분석
- 작성일: 2026-06-21

## 핵심 결론
소형 AI SaaS·GPT·웹앱의 핵심은 기능 수가 아니라 **반복 사용되는 좁은 문제, 사용자당 비용보다 높은 결제 가치, 장애·지원·보안을 감당하는 운영 구조**입니다.

## 성패를 가르는 것
데모에서 한 번 잘 작동하는 것과 매주 돈을 내고 쓰는 것은 다릅니다. 사용자가 기존 방식보다 얼마나 빠르고 정확하게 결과를 얻는지 측정해야 합니다. 무료 사용자가 늘수록 API 비용과 지원비도 커지므로 가입자 수보다 활성 사용자당 순수익과 재사용률이 중요합니다.

## 숨은 병목 5개
1. 흥미는 있지만 반복 사용하지 않음
2. 무료 사용량과 API 비용이 빠르게 증가함
3. 모델 오류를 제품 오류로 받아들임
4. 로그인·결제·데이터 저장·고객지원이 개발보다 커짐
5. 외부 모델·플랫폼 변경에 서비스가 의존함

## 숫자로 측정할 항목
가입 대비 첫 결과 완료율, 7일·30일 재사용률, 유료 전환율, 월 이탈률, 사용자당 API·서버비, 문의율, 장애시간, 환불률, 고객당 순수익을 기록합니다.

## 첫 수익과 실제 입금의 시간 차이
가입, 무료체험, 결제, 환불 가능 기간, 결제대행 정산, 실제 입금은 서로 다른 단계입니다. 구독 매출은 다음 달 유지 여부까지 봐야 합니다.

## 실패 사례
기능은 많지만 반복 문제가 아닌 경우, 무료 사용자가 늘어 비용만 증가하는 경우, 모델 변경으로 결과 품질이 흔들리는 경우, 맞춤 요청을 모두 받아 제품이 외주 프로젝트처럼 변하는 경우가 있습니다.

## 중단·전환 기준
재사용률이 낮으면 기능 추가보다 문제를 다시 정의합니다. 유료 전환 전 비용이 너무 크면 무료 한도를 줄입니다. 문의가 반복되면 입력 구조와 오류 안내를 고칩니다. 고객별 맞춤 개발이 늘면 SaaS와 구축 서비스를 분리합니다.

## 뻔한 조언으로 오해될 부분과 수정안
`빠르게 MVP를 만들어라`가 아니라 `한 사용자가 같은 문제로 다시 돌아오는 최소 기능을 만들어라`로 수정합니다.

## 공식 확인 사실
모델 API 가격, 무료 한도, 데이터 보관, 결제 수수료, 앱·GPT 배포 조건, 개인정보와 환불 정책은 공식 검증 전입니다.

## 추정 또는 편집 판단
7일·30일 재사용률과 사용자당 순수익을 핵심 지표로 보는 것은 사업성 판단입니다.

## 필요한 제품
LLM API, 인증, 데이터베이스, 호스팅, 결제, 분석, 오류 모니터링, 고객지원 도구를 조사해야 합니다.

## 오늘·3일·7일·30일
- 오늘: 한 사용자의 반복 문제와 기존 처리시간을 정합니다.
- 3일: 한 입력과 한 결과만 있는 프로토타입을 만듭니다.
- 7일: 실제 사용자에게 재사용 여부와 오류 지점을 확인합니다.
- 30일: 재사용률·전환율·API비·지원시간으로 계속할지 결정합니다.

## 위험과 주의사항
민감한 정보를 필요 이상 저장하지 않습니다. AI 결과를 확정적 판단으로 제시하지 않고 오류 안내와 사람 확인 경로를 둡니다. 결제·해지·데이터 삭제 절차를 명확히 해야 합니다.

## 총괄이 반드시 확인할 것
가입자 수보다 반복 사용, 사용자당 비용, 이탈률을 중심으로 설명하고 데모와 운영 제품의 차이를 분명히 해야 합니다.

- [ ] 인증·DB·호스팅 가격과 무료 한도
- [ ] 결제 수수료·구독·환불·정산 일정
- [ ] 입력 데이터 저장·학습·삭제 조건
- [ ] 개인정보·국외 이전·보안 관련 공식 기준
- [ ] 외부 모델 장애와 서비스 수준 조건
- [ ] 다음 원고에 반영할 API 단가, 무료 한도, 결제 수수료, 지급일
