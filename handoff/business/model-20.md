## 핵심 결론

소형 AI SaaS·GPT·웹앱은 기능을 많이 넣는 사업이 아닙니다. 성패는 **반복 사용되는 좁은 문제 × 첫 가치 도달 속도 × 사용자당 결제액과 AI·서버 비용의 차이 × 월 잔존·해지·결제 실패 × 장애·지원·보안 운영비 × 다채널 사례 유입의 효율**에서 갈립니다.

`활성 유료 사용자 × 사용자당 결제액 - AI·서버·PG·환불·지원·광고·개발비 = 월 순이익`

데모가 잘 작동해도 사용자가 매주 돌아오지 않거나 AI 호출비와 고객지원비가 결제액보다 크면 사업이 아닙니다. 첫 기능보다 반복 사용, 사용자당 비용 상한, 결제 실패, 데이터 삭제, 장애 대응을 먼저 설계해야 합니다.

## 성패를 가르는 것

### 1. 문제는 좁고 반복되어야 합니다

#### 관찰
“무엇이든 물어보는 AI”는 기존 도구와 차이가 약하고 결제 이유가 불분명합니다. 사용자가 한 번 신기해하고 떠나는 기능은 유료 구독을 만들기 어렵습니다.

#### 숨은 변수
- 월·주 사용 빈도
- 오류 한 건의 비용
- 현재 대안의 시간·가격
- 사용자가 입력할 원자료
- 결과가 다음 업무에 어떻게 쓰이는지
- 결과 저장·재사용 필요성

#### 작동 메커니즘
한 직무의 반복 문제를 짧은 입력과 구조화된 결과로 해결하면 재방문이 생깁니다. 결과가 보고서·견적·상담·콘텐츠·주문 처리로 바로 이어질수록 결제 가치가 높아집니다.

#### 실행 판단
제품 정의를 다음 한 문장으로 고정합니다.

`누가, 어떤 자료를 넣고, 어떤 결과를 얻어, 한 달에 몇 번 다시 쓰는가.`

초기에는 사용자 유형 1개와 핵심 작업 1개만 선택합니다.

#### 실패·전환 신호
- 첫 주 뒤 사용자가 돌아오지 않음
- 결과를 복사해도 다음 업무에서 쓰이지 않음
- 사용자가 범용 대화 기능만 요구함
- 문제 빈도가 월 1회 미만이고 대체재가 충분함

이 경우 기능 추가보다 문제·사용자·결제 방식을 바꿉니다.

### 2. 가입보다 첫 가치 도달을 측정해야 합니다

#### 관찰
회원가입과 무료체험은 사용자가 결과를 얻었다는 뜻이 아닙니다. 가입 뒤 빈 화면과 긴 설정이 나오면 유료 전환 전에 이탈합니다.

#### 숨은 변수
- 가입 단계 수
- 초기 데이터 입력량
- 첫 결과까지 걸린 시간
- 예시 데이터
- 오류 메시지와 복구
- 모바일·브라우저 차이

#### 작동 메커니즘
가입 직후 실제 업무 결과를 만들면 제품의 가치를 이해하고 다시 사용할 이유가 생깁니다. 반대로 첫 결과 전에 계정·설정·결제를 요구하면 체험 이탈이 커집니다.

#### 실행 판단
첫 화면에서 예시 입력과 결과를 보여주고 10분 안에 첫 결과가 나오게 설계합니다. 가입→첫 입력→첫 결과→저장→다시 사용을 단계별로 기록합니다.

#### 실패·전환 신호
- 가입은 늘지만 첫 결과 완료율이 낮음
- 예시 데이터로만 작동하고 실제 자료에서 실패함
- 오류 뒤 사용자가 복구하지 못함
- 첫 결과 뒤 저장·재사용이 없음

### 3. 사용자당 비용 상한을 먼저 정해야 합니다

#### 관찰
긴 문서·이미지·음성·반복 대화는 AI 호출비와 저장비를 빠르게 늘릴 수 있습니다. 무료 사용자가 많을수록 손실이 커질 수 있습니다.

#### 숨은 변수
- 입력·출력 길이
- 모델 선택
- 재시도와 중복 호출
- 캐시 가능성
- 파일 저장기간
- 무료 한도
- 악용·자동 호출
- 사용자당 최대 비용

#### 작동 메커니즘
기능별 평균비용과 최대비용을 측정하면 가격과 사용량 제한을 설계할 수 있습니다. 비용 상한이 없으면 사용자 증가가 매출보다 손실을 빠르게 키울 수 있습니다.

#### 실행 판단
기능마다 다음을 기록합니다.

`평균 입력량 → 평균 호출비 → 재시도율 → 저장비 → 사용자당 월 총비용`

저비용 모델·규칙·캐시·템플릿을 먼저 사용하고 고비용 처리는 추가 결제나 사용량 제한을 둡니다.

#### 실패·전환 신호
- 활성 사용자가 늘수록 사용자당 손실이 커짐
- 무료 한도를 반복 우회함
- 한 기능이 전체 비용 대부분을 차지함
- 사람이 오류를 수정하느라 지원비가 급증함

### 4. 유료 전환보다 잔존·해지·결제 실패를 봅니다

#### 관찰
첫 달 할인과 무료체험은 결제를 만들 수 있지만 반복 가치가 없으면 다음 달 해지됩니다. 카드 만료와 결제 실패도 매출 누수를 만듭니다.

#### 숨은 변수
- 첫 결제 이유
- 주·월 사용 빈도
- 월 잔존
- 해지 이유
- 결제 실패와 재시도
- 환불
- 연간 결제 할인
- 재활성 사용자

#### 작동 메커니즘
사용자가 결과를 저장하고 반복 업무에 재사용할 때 잔존이 높아집니다. 신규 결제만 늘리고 해지·실패를 관리하지 않으면 광고비만 커집니다.

#### 실행 판단
월별로 다음 집단을 구분합니다.

- 신규 가입
- 첫 결과 완료
- 주·월 활성
- 첫 결제
- 잔존
- 해지
- 결제 실패
- 재활성

#### 실패·전환 신호
- 신규 결제와 해지가 같은 속도로 늘어남
- 무료체험 종료 직후 해지가 집중됨
- 결제 실패 재시도 절차가 없음
- 월 사용 빈도가 낮아 구독 이유가 약함

이 경우 광고를 늘리지 않고 가격·결제 방식·핵심 작업을 바꿉니다.

### 5. 고객지원과 장애가 숨은 제품 기능입니다

#### 관찰
AI 결과 오류, 로그인, 결제, 파일 업로드, 브라우저 차이, 외부 API 장애로 문의가 발생합니다. 작은 제품도 운영자가 모든 문제를 직접 받으면 확장되지 않습니다.

#### 숨은 변수
- 문의 유형과 빈도
- 응답시간
- 오류 로그
- 장애 탐지와 공지
- 환불 기준
- 셀프 도움말
- 지원시간 상한

#### 작동 메커니즘
로그·오류 메시지·도움말이 있으면 사용자가 스스로 복구할 수 있습니다. 장애와 환불 기준이 없으면 고객 한 명당 지원시간이 누적됩니다.

#### 실행 판단
오류 코드, 사용 로그, 상태 안내, FAQ, 문의 양식, 환불·장애 보상 기준을 준비합니다. 반복 문의는 기능·문구·온보딩 수정으로 전환합니다.

#### 실패·전환 신호
- 사용자당 지원시간이 줄지 않음
- 같은 오류 문의가 반복됨
- 장애 원인을 찾을 로그가 없음
- 지원시간이 사용자당 결제액을 넘음

### 6. 개인정보·권한·AI 책임을 분리해야 합니다

#### 관찰
사용자가 계약서·상담·고객·주문·사내 자료를 입력할 수 있습니다. 제품이 작아도 민감정보 위험은 작지 않습니다.

#### 숨은 변수
- 수집 목적과 최소 항목
- 보관기간과 삭제
- 외부 API 전송
- 관리자 접근
- 모델 학습 사용 여부
- 팀 권한
- 사고 대응

#### 작동 메커니즘
데이터 흐름을 설명하지 못하면 신뢰·계약·법적 위험이 생깁니다. 민감정보가 필요 없는 기능이라면 입력 자체를 막는 것이 비용이 가장 낮습니다.

#### 실행 판단
`수집 → 외부 전송 → 처리 → 저장 → 조회 → 백업 → 삭제`를 문서화합니다. 초기 제품은 민감정보 입력 경고, 최소 보관, 역할별 권한, 삭제 요청 절차를 둡니다.

#### 실패·전환 신호
- 데이터 위치와 삭제방법을 설명하지 못함
- 관리자가 모든 사용자 자료를 볼 수 있음
- 민감정보를 기능과 무관하게 수집함
- 사고 대응과 백업 복구 기준이 없음

### 7. 제품 시연은 기능 소개보다 반복 문제와 결과를 보여줘야 합니다

#### 관찰
화려한 AI 화면은 조회를 만들 수 있지만 구매자는 자신의 업무에서 어떤 시간을 줄이고 어떤 결과를 얻는지 알고 싶어 합니다.

#### 숨은 변수
- 공개 가능한 예시 데이터
- 전후 시간·오류 수치의 근거
- 채널별 첫 장면과 길이
- 무료체험·가입·결제 링크의 마찰
- 사례 콘텐츠 제작시간
- 유입별 유료 전환

#### 작동 메커니즘
한 반복 문제의 입력·처리·결과·다시 사용을 짧게 보여주면 구매자가 제품의 역할을 이해합니다. 같은 원본을 채널별 이용 방식에 맞게 바꾸면 유입비용을 분산할 수 있습니다.

#### 실행 판단
한 제품 사례를 다음처럼 재가공합니다.

- 유튜브: 전체 문제·사용법·비용·한계
- 인스타그램: 전후 화면·저장 카드·프로필 이동
- 틱톡: 한 작업의 빠른 입력·결과·댓글 답변
- X: 짧은 영상과 핵심 수치·개발 기록
- 페이스북: 업종·지역 그룹의 문제 사례와 체험 안내

CapCut으로 자동 자막·다중 플랫폼 편집을 먼저 확인합니다.

#### 실패·전환 신호
- 영상 조회는 늘지만 첫 결과 완료가 늘지 않음
- 가입은 늘지만 유료 전환·잔존이 낮음
- 채널별 제작시간이 유료 고객 이익을 넘음
- 제품의 한계와 데이터 조건을 숨겨 환불이 늘어남

## 핵심 현실 인사이트

- 구체적 관찰: 가입·무료체험·첫 결제·활성 사용·잔존·순이익은 서로 다른 사건입니다.
- 남들이 놓치는 숨은 변수: 첫 결과시간, 사용자당 최대 AI 비용, 결제 실패, 지원시간, 데이터 삭제, 유입별 잔존입니다.
- 결과가 달라지는 작동 메커니즘: 반복 문제를 좁히고 사용량·비용·오류를 통제하면 사용자가 늘어도 마진이 남습니다.
- 임계점·시간차·교환비용: `콘텐츠·광고 → 가입 → 첫 결과 → 유료 결제 → 반복 사용 → 해지·환불·결제 실패 → PG 정산 → 원화 계좌 입금`
- 초보자가 먼저 측정하거나 결정할 것: 문제 빈도, 첫 결과, 사용자당 비용 상한, 결제 방식, 지원 범위입니다.
- 계속할 조건: 활성·잔존이 개선되고 결제액에서 AI·서버·PG·지원비를 뺀 이익이 남는 경우입니다.
- 중단·전환 신호: 가입만 늘고 첫 결과·재사용이 없거나 사용자가 늘수록 손실이 커지는 경우입니다.
- 대표 사례·수치·반례: 월 1회 쓰는 범용 챗봇보다 매주 반복되는 견적·보고·분류 도구가 더 높은 결제 가치를 만들 수 있습니다.
- 반복 확인된 구조인지 단일 사례의 편집 판단인지: 활성·잔존·사용자당 비용을 우선하는 것은 사업성 판단이며 PG·API·개인정보 조건은 공식 확인이 필요합니다.

## 시장 현실

- 네이버 블로그·카페·브런치·유튜브·커뮤니티에서 먼저 확인한 동향: 범용 AI보다 견적·보고·콘텐츠·상담·주문처럼 한 업무를 끝내는 작은 도구가 이해와 판매가 쉽습니다.
- 반복된 패턴: 가입자·기능·데모 화면을 강조하고 잔존·AI 비용·지원·결제 실패를 공개하지 않는 사례가 많습니다.
- 상충되는 의견: 월 구독과 사용량·건별 결제 중 어느 방식이 유리한지는 실제 사용 빈도와 비용 구조에 따라 달라집니다.
- 작성자 이해관계 또는 광고 가능성: 개발 도구·호스팅·강의 판매자는 출시 속도를 강조할 유인이 있습니다.
- 이미 포화된 부분: 범용 챗봇, 단순 프롬프트 포장, 저장·업무연결이 없는 일회성 생성기입니다.
- 아직 남은 기회: 한글 문서·원화 결제·실제 업무 데이터·저장·재사용·세팅 지원을 결합한 좁은 도구입니다.

## 판매·수익·광고 경로

- 실제 구매자·독자·시청자·광고주: 소상공인, 1인 사업자, 프리랜서, 교육·상담·쇼핑몰·기관 실무자
- 발견·유입 경로: 네이버 검색·블로그, 유튜브·인스타그램·틱톡·X·페이스북 사례 콘텐츠, 당근, 크몽, 위시켓, 기존 고객

- 언어 지원·지역 제한: 한글 입력·출력·고객지원과 실제 서비스 가능 지역을 기준으로 합니다.

- 적용 확인이 필요한 항목: 정기결제, 개인정보, 외부 API 전송, 짧은 영상 링크·광고 표시, 환불·장애 기준입니다.
