## 핵심 결론

AI 업무자동화 구축 대행·컨설팅은 도구를 많이 연결하는 사업이 아닙니다. 성패는 **고객의 실제 병목을 수치로 정의하는 진단 × 예외와 실패를 포함한 운영 설계 × 개인정보·권한 통제 × 구축 후 유지보수까지 남는 계약 구조 × 사례를 다채널 콘텐츠로 전환하는 영업 구조**에서 갈립니다.

`고객의 절감가치 - 구축비·API·서버·교육·장애·유지보수 비용 = 고객가치`

`계약금액 - 진단·설계·개발·시험·회의·지원·외주·세금 비용 = 공급자 순이익`

자동화 데모가 잘 작동해도 현장 데이터가 불완전하거나 직원이 우회하면 사용되지 않습니다. 구축비만 계산하고 요구사항 변경·장애 대응·사용자 교육·API 비용·유지보수 시간을 빼면 실제 마진을 잘못 읽게 됩니다.

## 성패를 가르는 것

### 1. 자동화 전에 기준 업무를 측정해야 합니다

#### 관찰
고객은 “AI로 자동화해 달라”고 요청하지만 현재 처리시간·오류·담당자·승인 단계가 정리되지 않은 경우가 많습니다.

#### 숨은 변수
- 월 처리 건수
- 건당 처리시간
- 오류·재작업률
- 승인자 수와 대기시간
- 업무가 몰리는 시간
- 현재 도구와 수작업 우회
- 오류 한 건의 비용

#### 작동 메커니즘
자동화 전 기준값이 있어야 절감효과와 가격을 설명할 수 있습니다. 기준값이 없으면 성공 여부도 판단할 수 없어 고객과 공급자가 서로 다른 결과를 기대합니다.

#### 실행 판단
실제 작업 20~50건을 관찰해 다음을 기록합니다.

`입력 → 판단 → 승인 → 출력 → 보관 → 예외 → 재작업`

진단 보고서에는 현재시간, 목표시간, 오류비용, 자동화 금지 구간을 적습니다.

#### 실패·전환 신호
- 기준 시간과 오류율을 측정할 수 없음
- 담당자마다 업무 방식이 다름
- 입력자료 형식이 매번 바뀜
- 고객이 자동화 뒤의 책임자를 정하지 못함

이 경우 개발 계약이 아니라 유료 진단·업무정의 프로젝트로 전환합니다.

### 2. 정상 흐름보다 예외 처리가 품질을 결정합니다

#### 관찰
데모는 정상 데이터로 작동하지만 현장에는 누락·중복·오타·권한 부족·긴급 처리·외부 시스템 장애가 있습니다.

#### 숨은 변수
- 예외 발생률
- 수동 전환 기준
- 재실행 가능 여부
- 오류 로그
- 알림 수신자
- 복구시간
- 중복 실행 방지

#### 작동 메커니즘
예외를 사람이 안전하게 처리할 수 있어야 자동화가 업무를 멈추지 않습니다. 실패 원인을 찾지 못하면 작은 오류가 데이터 누락과 고객 분쟁으로 확대됩니다.

#### 실행 판단
- 예외 목록과 우선순위를 만듭니다.
- 실패 시 수동 처리 화면을 둡니다.
- 재실행·중복 방지·상태 로그를 설계합니다.
- 장애 알림과 책임자를 정합니다.

#### 실패·전환 신호
- 실패 원인을 로그에서 찾을 수 없음
- 재실행 시 중복 데이터가 생김
- 수동 복구가 원래 업무보다 오래 걸림
- 외부 서비스 장애가 전체 업무를 멈춤

### 3. AI 판단과 사람 승인을 오류비용에 따라 나눕니다

#### 관찰
분류·요약·초안은 빠르지만 계약·가격·인사·민원·상담 같은 판단은 오류 비용이 큽니다.

#### 숨은 변수
- 오답 한 건의 비용
- 근거 추적 가능성
- 승인 대기시간
- 책임 소재
- 민감정보 포함 여부
- 사람이 다시 확인하는 비율

#### 작동 메커니즘
낮은 위험은 자동 처리하고 높은 위험은 근거와 함께 사람에게 넘기면 안정성이 높아집니다. 사람이 모든 결과를 다시 확인한다면 AI는 시간을 줄이지 못합니다.

#### 실행 판단
업무를 세 등급으로 나눕니다.

1. 자동 실행 가능
2. 사람 승인 뒤 실행
3. 자동화 금지

AI 결과에 근거·원문·신뢰도·검토 항목을 함께 보여줍니다.

#### 실패·전환 신호
- 사람이 결과를 전부 다시 작성함
- 오류 근거를 설명할 수 없음
- 승인 대기가 자동화 절감시간보다 큼
- 높은 위험 업무가 자동 실행됨

### 4. 데이터·권한·개인정보가 기술보다 먼저입니다

#### 관찰
고객 파일·메일·주문·상담·직원 데이터에는 개인정보와 영업정보가 포함될 수 있습니다.

#### 숨은 변수
- 수집 목적과 최소 항목
- 보관기간과 삭제
- 외부 API 전송
- 역할별 접근권한
- 관리자 로그
- 퇴사자 계정 회수
- 백업과 사고 대응

#### 작동 메커니즘
데이터 흐름과 책임을 설명하지 못하면 작은 자동화도 계약·보안 위험이 됩니다. 접근권한이 과도하면 내부 사고 가능성도 커집니다.

#### 실행 판단
`수집 → 전송 → 처리 → 저장 → 조회 → 백업 → 삭제` 흐름을 문서화합니다. 비밀정보 마스킹, 역할별 권한, 계정 회수, 사고 통지 기준을 정합니다.

#### 실패·전환 신호
- 고객이 데이터 위치와 접근자를 모름
- 외부 API에 어떤 정보가 전송되는지 설명하지 못함
- 삭제·백업·복구 절차가 없음
- 하나의 관리자 계정을 여러 사람이 공유함

이 경우 데이터 연결을 중단하고 범위를 줄입니다.

### 5. 구축비보다 변경·유지보수 계약이 사업성을 좌우합니다

#### 관찰
고객 업무·API·화면·조직이 바뀌면 자동화도 수정해야 합니다. 하자와 기능 추가를 구분하지 않으면 무상 지원이 계속됩니다.

#### 숨은 변수
- 무상 하자보수 기간
- 고객 요청 변경
- 외부 API 변경
- 월 점검
- 긴급 장애 대응시간
- 교육과 인수인계
- 운영 중단 시 데이터 회수

#### 작동 메커니즘
변경 범위를 계약하지 않으면 구축 후 지원시간이 누적됩니다. 월 유지보수료가 있어도 지원시간 상한이 없으면 마진이 사라질 수 있습니다.

#### 실행 판단
계약에서 다음을 구분합니다.

- 하자 수정
- 기능 추가
- 고객 업무 변경
- 외부 서비스 변경
- 월 점검
- 긴급 대응
- 종료·인수인계

#### 실패·전환 신호
- 월 지원시간이 유지보수료를 넘음
- 고객별 구조가 지나치게 달라짐
- 장애 원인이 외부 서비스인데 무상 대응이 반복됨
- 인수인계 자료가 없어 공급자에게 계속 의존함

### 6. 사례 콘텐츠는 기술 설명보다 전후 비용을 보여줘야 합니다

#### 관찰
자동화 구축 영업에서 화려한 화면은 관심을 만들지만 구매자는 실제 절감시간·오류 감소·운영 책임을 알고 싶어 합니다.

#### 숨은 변수
- 공개 가능한 고객 정보
- 전후 비교 숫자의 근거
- 짧은 영상에서 보여줄 수 있는 업무 단계
- 채널별 첫 문장과 행동 유도
- 사례 제작시간
- 문의의 질

#### 작동 메커니즘
“AI를 연결했다”보다 “월 300건의 입력 업무에서 어떤 단계와 오류를 줄였는가”를 보여주면 구매자가 자신의 업무와 연결하기 쉽습니다.

#### 실행 판단
하나의 익명화된 사례에서 다음 콘텐츠를 만듭니다.

- 유튜브: 전체 업무 흐름·전후 비용·예외 처리 설명
- 인스타그램: 전후 화면·체크리스트·프로필 문의
- 틱톡: 반복 업무 한 장면과 절감 포인트
- X: 핵심 수치·구조·실패 조건을 짧은 영상과 글로 연결
- 페이스북: 업종·지역 그룹의 문제 중심 사례와 상담 안내

CapCut의 자동 자막·다중 플랫폼 편집을 먼저 확인하고, 사례별 개인정보·비밀유지 범위를 계약서와 대조합니다.

#### 실패·전환 신호
- 영상 조회는 늘지만 유효 상담이 없음
- 고객 정보가 과도하게 노출됨
- 전후 수치의 측정 근거가 없음
- 다채널 운영시간이 계약 이익보다 커짐

## 핵심 현실 인사이트

- 구체적 관찰: 자동화 성공은 데모가 아니라 현장 사용률·예외 복구·절감시간으로 결정됩니다.
- 남들이 놓치는 숨은 변수: 기준 업무 측정, 예외율, 전면 재검수, 권한, 유지보수 부채, 영업 사례의 근거입니다.
- 결과가 달라지는 작동 메커니즘: 정상·예외·승인·로그·복구를 함께 설계하면 업무가 멈추지 않습니다.
- 임계점·시간차·교환비용: `문의 → 진단 계약 → 착수금 → 설계·구축 → 시험운영 → 검수·인수 → 잔금 → 유지보수 정산 → 계좌 입금`
- 초보자가 먼저 측정하거나 결정할 것: 월 처리량, 건당 시간, 오류비용, 자동화 금지 범위, 유지보수 상한입니다.
- 계속할 조건: 고객의 절감효과가 유지비보다 크고 공급자의 지원시간 차감 순이익이 남는 경우입니다.
- 중단·전환 신호: 현장 사용률이 낮거나 사람이 결과를 전부 재검수하거나 지원시간이 계약금을 넘는 경우입니다.
- 대표 사례·수치·반례: 5분 작업을 자동화해도 하루 2건이면 가치가 작고, 30초 작업이라도 하루 수천 건이면 가치가 클 수 있습니다.
- 반복 확인된 구조인지 단일 사례의 편집 판단인지: 기준 업무·예외·유지보수를 함께 보는 것은 사업성 판단이며 플랫폼별 계약·보안 조건은 공식 확인이 필요합니다.

## 시장 현실

- 네이버 블로그·카페·브런치·유튜브·커뮤니티에서 먼저 확인한 동향: 생성형 AI 데모보다 주문·상담·보고·문서 처리처럼 반복 업무를 줄이는 소형 자동화 수요가 많습니다.
- 반복된 패턴: 도구 연결 화면과 “완전자동화”를 강조하고 예외·보안·유지보수·사용자 교육을 과소평가하는 사례가 반복됩니다.
- 상충되는 의견: 맞춤 구축과 표준 패키지 중 어느 쪽이 유리한지는 고객 업무의 예외와 반복성에 따라 달라집니다.
- 작성자 이해관계 또는 광고 가능성: 자동화 도구·강의·템플릿 판매자는 구현 난도를 낮게 보일 유인이 있습니다.
- 이미 포화된 부분: 범용 챗봇, 단순 알림 연결, 근거 없는 자동 응답 데모입니다.
- 아직 남은 기회: 한 업종의 주문·상담·견적·보고·분류를 예외와 승인까지 포함해 표준화한 패키지입니다.

## 판매·수익·광고 경로

- 실제 구매자·독자·시청자·광고주: 소상공인, 쇼핑몰 운영자, 교육기관, 병원·상담기관, 제조·물류·재단·기업 실무팀
- 발견·유입 경로: 네이버 검색·블로그, 유튜브·인스타그램·틱톡·X·페이스북 사례 콘텐츠, 당근, 크몽, 숨고, 위시켓, 기존 고객 소개
- 판매 또는 광고가 일어나는 플랫폼: 유료 진단·직접 계약, 크몽·숨고·위시켓, 당근 비즈프로필·지역광고
- 결제수단: 플랫폼 대금보호, 계좌이체, 신용카드·간편결제, 세금계산서 조건의 결제
- 수익 지급자: 플랫폼 또는 고객
- 플랫폼 수수료·광고비·환불·CS 비용: 플랫폼 수수료, 네이버 검색광고, Meta 광고, 진단·개발·API·서버·교육·장애대응·외주·보안·세금 비용
- 확정 수익 조건: 진단·구축·검수·인수 조건을 통과하고 환불·분쟁·대금 조건이 확정된 경우
- 실제 입금 시점과 경로: 착수금·중도금·잔금과 월 유지보수료가 플랫폼 또는 원화 계좌로 입금
- 반복 가능한 구조인지: 공통 업무를 표준 패키지로 만들고 유지보수 시간이 통제되어야 합니다.
- 대상 범위 교정: 실제 고객·한글 업무·원화 결제·원화 정산을 기준으로 합니다.

## 공식 확인 사실

사업성 분석 단계에서는 크몽·숨고·위시켓의 계약·대금보호·수수료·분쟁·정산, 당근의 홍보·외부 링크, 개인정보 처리·외부 API 전송·클라우드 보안·전자계약·세금서류 조건을 확정하지 않습니다. 다음 단계가 공식 문서로 확인해야 합니다.

## 현실 기준 교정

- 미국 기준 자료인지: 대응되는 구매자·계약·결제·정산이 없는 외부 사례는 실행 모델에서 제외합니다.
- 판매자 형태별 적용 가능 여부: 개인과 개인사업자의 컨설팅·개발 계약·세금계산서·원천징수 조건을 확인합니다.
- 결제·지급·정산 조건: 진단·구축·유지보수의 착수금·중도금·잔금·대금보호를 구분합니다.
- 원화 가격·세금·수수료: 플랫폼 수수료·API·서버·외주·부가가치세·종합소득세를 포함합니다.
- 언어 지원·지역 제한: 한글 문서·상담·업무 흐름과 실제 방문·원격 지원 범위를 기준으로 합니다.
- 플랫폼·소비 습관 차이: 네이버·당근·크몽·숨고·위시켓의 문의·계약 방식이 같다고 가정하지 않습니다.
- 적용 확인이 필요한 항목: 개인정보·외부 API·로그·백업·계약·다채널 사례 공개 조건입니다.

## 추정 또는 편집 판단

- 첫 단계에서 진단을 유료로 분리하고 자동화 등급을 세 단계로 나누는 것은 초기 운영 기준안입니다.
