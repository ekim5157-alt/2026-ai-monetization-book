# HANDOFF

## 핵심 결론
이 모델의 핵심은 AI 도구를 연결하는 기술이 아니라 **고객의 실제 병목을 수치로 정의하는 진단, 예외와 실패를 포함한 운영 설계, 구축 후 유지보수까지 남는 계약 구조**입니다.

## 성패를 가르는 것
자동화할 업무를 잘못 고르면 기술적으로 성공해도 비용을 줄이지 못합니다. 반복 횟수, 처리시간, 오류비용, 승인 단계, 예외 비율을 먼저 측정해야 합니다. 데모와 실제 운영은 다르며, 권한·데이터·사람의 승인·복구 절차가 없으면 현장에 정착하지 못합니다.

## 숨은 병목 5개
1. 고객 업무가 문서화되어 있지 않음
2. 예외 상황이 표준 흐름보다 많음
3. 기존 시스템의 권한·API·데이터 품질 문제
4. 구축 후 수정과 장애 대응이 무상 범위로 늘어남
5. 담당자 교체와 현장 저항으로 사용이 중단됨

## 숫자로 측정할 항목
자동화 전후 처리시간, 건당 비용, 오류율, 사람의 검수시간, 예외 처리율, 도입률, 장애 복구시간, 월 유지시간, 프로젝트 순이익, 재계약률을 기록합니다.

## 첫 수익과 실제 입금의 시간 차이
진단, 제안, 계약, 선금, 파일럿, 검수, 운영 전환, 잔금, 유지보수비 입금은 서로 다른 단계입니다. 계약금액과 실제 현금흐름을 분리해야 합니다.

## 실패 사례
고객이 원하는 도구부터 구축했지만 실제 병목이 아닌 경우, 데모는 되지만 예외와 권한 문제로 운영되지 않는 경우, 범위가 계속 늘어 손실이 나는 경우, 담당자가 사용하지 않아 자동화가 폐기되는 경우가 있습니다.

## 중단·전환 기준
절감할 시간과 비용이 구축·유지비보다 작으면 자동화를 중단합니다. 예외율이 높으면 완전자동화 대신 사람 승인형으로 전환합니다. 데이터 품질이 낮으면 AI보다 입력 표준화부터 합니다. 무상 수정이 반복되면 운영 계약과 변경요청 절차를 분리합니다.

## 뻔한 조언으로 오해될 부분과 수정안
`반복 업무를 자동화하라`가 아니라 `반복 횟수·예외율·오류비용을 측정해 투자 회수가 되는 업무만 자동화하라`로 수정합니다.

## 공식 확인 사실
API 가격, 사용량 한도, 데이터 보관, 개인정보, 외부 서비스 연동, 클라우드 비용, 계약과 세금 조건은 공식 검증 전입니다.

## 추정 또는 편집 판단
예외율과 사람 검수시간, 도입률을 핵심 지표로 보는 것은 사업성 판단입니다.

## 필요한 제품
LLM API, 자동화 플랫폼, 데이터베이스, 클라우드, 모니터링, 프로젝트 관리, 계약·결제 도구의 가격과 조건을 조사해야 합니다.

## 오늘·3일·7일·30일
- 오늘: 한 업무의 반복 횟수·시간·오류·예외를 측정합니다.
- 3일: 자동화 전후 흐름과 사람 승인 지점을 그립니다.
- 7일: 작은 유료 파일럿으로 실제 데이터를 시험합니다.
- 30일: 절감액·오류·유지시간으로 확대 여부를 결정합니다.

## 위험과 주의사항
고객의 비공개 데이터와 계정 권한을 최소한으로 사용해야 합니다. 자동 결과를 무조건 실행하지 않고 로그, 승인, 복구 절차를 둬야 합니다.

## 총괄이 반드시 확인할 것
기술 기능보다 업무 진단, 예외 처리, 현장 도입, 유지보수 원가를 중심으로 설명해야 합니다.
