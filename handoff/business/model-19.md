## 핵심 결론

AI 업무자동화 구축 대행·컨설팅은 도구를 많이 연결하는 사업이 아닙니다. 성패는 **고객의 실제 병목을 수치로 정의하는 진단 × 예외와 실패를 포함한 운영 설계 × 개인정보·권한 통제 × 구축 후 유지보수까지 남는 계약 구조**에서 갈립니다.

`절감 가능한 반복시간 × 오류 감소 × 실제 사용률 × 예외 처리 안정성 × 구축·유지비 차감 후 고객가치 × 공급자 순이익`

시연 화면이 잘 작동해도 현장 데이터가 불완전하거나 직원이 우회하면 자동화는 쓰이지 않습니다. 구축비뿐 아니라 요구사항 변경·장애 대응·사용자 교육·API 비용·유지보수 시간을 원가로 계산해야 합니다.

## 성패를 가르는 것

### 자동화보다 기준 업무를 먼저 측정합니다
- 관찰: 고객은 “AI로 자동화해 달라”고 요청하지만 현재 처리시간·오류·담당자가 정리되지 않은 경우가 많습니다.
- 숨은 변수: 월 처리 건수, 건당 시간, 오류율, 승인 단계, 재작업, 병목 담당자입니다.
- 작동 메커니즘: 자동화 전 기준값이 있어야 절감효과와 가격을 설명할 수 있습니다.
- 실행 판단: 작업 20~50건을 관찰해 입력·판단·승인·출력·예외를 흐름도로 기록합니다.
- 실패·전환 신호: 기준 시간과 오류를 측정할 수 없으면 개발보다 진단 프로젝트로 전환합니다.

### 정상 흐름보다 예외 처리가 품질을 결정합니다
- 관찰: 데모는 정상 데이터로 작동하지만 현장에는 누락·중복·오타·권한·긴급 처리 예외가 있습니다.
- 숨은 변수: 예외율, 수동 전환 기준, 재처리, 로그, 승인자, 알림입니다.
- 작동 메커니즘: 예외를 사람이 안전하게 처리할 수 있어야 자동화가 업무를 멈추지 않습니다.
- 실행 판단: 예외 목록, 수동 처리 화면, 재실행, 실패 알림, 책임자를 설계합니다.
- 실패·전환 신호: 자동화 실패 뒤 원인을 찾지 못하거나 수동 복구가 오래 걸리면 범위를 줄입니다.

### AI가 판단할 범위와 사람이 승인할 범위를 나눕니다
- 관찰: 분류·요약·초안은 빠르지만 계약·가격·의료·인사·민원 같은 판단은 오류 비용이 큽니다.
- 숨은 변수: 오답 비용, 근거 추적, 승인 지연, 책임 소재, 데이터 민감도입니다.
- 작동 메커니즘: 낮은 위험은 자동 처리하고 높은 위험은 근거와 함께 사람에게 넘기면 안정성이 높아집니다.
- 실행 판단: 자동 실행·승인 필요·자동화 금지의 세 등급으로 업무를 나눕니다.
- 실패·전환 신호: 사람이 결과를 전부 다시 확인한다면 AI 단계가 시간 절감이 아니라 추가 업무가 된 것입니다.

### 데이터·권한·개인정보가 기술보다 먼저입니다
- 관찰: 고객 파일·메일·주문·상담 데이터에는 개인정보와 영업정보가 포함될 수 있습니다.
- 숨은 변수: 수집 목적, 보관기간, 접근권한, 외부 API 전송, 로그, 계정 퇴사 처리입니다.
- 작동 메커니즘: 권한과 삭제 절차가 없으면 작은 자동화도 보안·계약 위험이 됩니다.
- 실행 판단: 최소 데이터, 역할별 권한, 비밀정보 마스킹, 로그·백업·삭제 기준을 문서화합니다.
- 실패·전환 신호: 고객이 데이터 위치·접근자·삭제방법을 설명하지 못하면 연결을 중단합니다.

### 구축비보다 변경·유지보수 계약이 사업성을 좌우합니다
- 관찰: 고객 업무·API·화면·조직이 바뀌면 자동화도 수정해야 합니다.
- 숨은 변수: 무상 하자보수, 기능 추가, 월 점검, 장애 대응시간, 외부 서비스 비용, 인수인계입니다.
- 작동 메커니즘: 변경 범위를 계약하지 않으면 구축 후 무상 지원이 계속됩니다.
- 실행 판단: 하자·변경·운영지원·긴급대응을 구분하고 월 유지보수 또는 건별 단가를 정합니다.
- 실패·전환 신호: 유지보수 시간이 월 계약금보다 크거나 고객별 구조가 지나치게 달라지면 표준 패키지를 줄입니다.

## 핵심 현실 인사이트
- 구체적 관찰: 자동화 성공은 데모가 아니라 현장 사용률·예외 복구·절감시간으로 결정됩니다.
- 남들이 놓치는 숨은 변수: 기준 업무 측정, 예외율, 전면 재검수, 권한, 유지보수 부채입니다.
- 결과가 달라지는 작동 메커니즘: 정상·예외·승인·로그·복구를 함께 설계하면 업무가 멈추지 않습니다.
- 임계점·시간차·교환비용: `문의 → 진단 계약 → 착수금 → 설계·구축 → 시험운영 → 검수·인수 → 잔금 → 유지보수 정산 → 계좌 입금`
- 초보자가 먼저 측정하거나 결정할 것: 월 처리량, 건당 시간, 오류비용, 자동화 금지 범위, 유지보수 상한입니다.
- 계속할 조건: 고객의 절감효과가 유지비보다 크고 공급자의 지원시간 차감 순이익이 남는 경우입니다.
- 중단·전환 신호: 현장 사용률이 낮거나 사람이 결과를 전부 재검수하거나 지원시간이 계약금을 넘는 경우입니다.
- 대표 사례·수치·반례: 5분 작업을 자동화해도 하루 2건이면 가치가 작고, 30초 작업이라도 하루 수천 건이면 가치가 클 수 있습니다.
- 반복 확인된 구조인지 단일 사례의 편집 판단인지: 기준 업무·예외·유지보수를 함께 보는 것은 사업성 판단이며 플랫폼별 계약·보안 조건은 공식 확인이 필요합니다.

## 첫 수익과 실제 입금의 시간 차이

`네이버·당근·크몽·숨고·위시켓·소개 유입 → 진단·견적 → 계약·착수금 → 설계·구축 → 시험운영·인수 → 잔금·세금서류 → 월 유지보수 → 원화 계좌 입금`

## 시장 현실

- 구매자: 소상공인, 쇼핑몰 운영자, 교육기관, 병원·상담기관, 제조·물류·재단·기업 실무팀
- 유입: 네이버 검색·블로그, 당근, 크몽, 숨고, 위시켓, 기존 고객 소개
- 판매 방식: 진단 컨설팅, 소형 구축 패키지, 프로젝트 계약, 월 유지보수
- 결제수단: 플랫폼 대금보호, 계좌이체, 신용카드·간편결제, 세금계산서 조건의 결제
- 수익 지급자: 플랫폼 또는 고객
- 비용: 조사·회의·개발·API·서버·시험·교육·장애대응·외주·보안·세금
- 반복 가능성: 공통 업무를 표준 패키지로 만들고 유지보수 시간이 통제되어야 합니다.

## 판매·수익·광고 경로

- 사례·진단 콘텐츠·당근 지역 노출 → 상담 → 유료 진단 → 구축 → 유지보수
- 네이버 검색광고는 “주문 자동화”, “상담 분류”, “보고서 자동화”처럼 구체적인 문제 검색어에 한정해 시험합니다.
- 당근은 지역 소상공인의 반복 업무 진단·방문 상담 유입 경로로 검토하고 외부 링크·서비스 홍보 정책을 확인합니다.

## 공식 확인 사실

사업성 분석 단계에서는 크몽·숨고·위시켓의 계약·대금보호·수수료·분쟁·정산, 당근 홍보 정책, 개인정보 처리·외부 API 전송·클라우드 보안·전자계약·세금서류 조건을 확정하지 않습니다. 다음 단계가 공식 문서로 확인해야 합니다.

## 현실 기준 교정

- 개인과 개인사업자의 컨설팅·개발 계약·세금계산서·원천징수 조건을 확인합니다.
- 카카오는 독립 수익 플랫폼이 아니라 고객 연락·공지·인증 연동 수단으로만 봅니다.
- 고객가치와 공급자 순이익을 각각 계산합니다.
- AI가 내린 결정을 무조건 자동 실행하지 않고 오류비용에 따라 사람 승인을 둡니다.

## 추정 또는 편집 판단

- 첫 단계에서 진단을 유료로 분리하고 자동화 등급을 세 단계로 나누는 것은 초기 운영 기준안입니다.
- 5분 작업과 30초 작업의 예시는 빈도와 절감시간의 관계를 설명하기 위한 반례입니다.

`크몽 — 컨설팅·개발 상품 등록·수수료·환불·정산 조건 확인.`

## 오늘·3일·7일·30일

- 오늘: 반복 업무 1개와 월 처리량·시간·오류를 측정합니다.
- 3일: 정상 흐름·예외·승인·로그·복구를 포함한 진단서를 만듭니다.
- 7일: 작은 데이터로 시험운영하고 자동·승인·수동 비율을 기록합니다.
- 30일: 절감시간·오류·사용률·지원시간·고객가치·공급자 순이익을 비교합니다.

## 위험과 주의사항

개인정보·영업정보를 최소 수집하고 외부 전송·보관·삭제를 계약에 명시합니다. 무단 계정 공유, 근거 없는 완전자동화 약속, 의사결정 책임 회피, 로그·백업 없는 운영을 피합니다.

## 총괄이 반드시 확인할 것

1. 데모와 현장 사용률·절감효과를 구분했는가.
2. 정상 흐름·예외·승인·로그·복구를 다뤘는가.
3. 구축비와 유지보수·장애대응 원가를 분리했는가.
4. 당근을 지역 진단·상담 유입 경로로만 다뤘는가.
5. 플랫폼·결제·정산·세금·보안 경로가 구체적인가.
6. 문서 마지막에 조사 체크리스트가 있는가.
