## 핵심 결론

AI 업무자동화 구축 대행·컨설팅은 도구를 많이 연결하는 사업이 아닙니다. 성패는 **현재 업무의 병목을 수치로 정의하는 진단 × 예외와 실패를 포함한 운영 설계 × 사람 승인과 책임 경계 × 개인정보·권한 통제 × 구축 뒤 유지보수까지 남는 계약 구조**에서 갈립니다.

`고객의 절감가치 - 구축비·API·서버·교육·장애·유지보수 비용 = 고객가치`

`계약금액 - 진단·설계·개발·시험·회의·지원·외주·세금 비용 = 공급자 순이익`

데모가 작동해도 현장 데이터가 불완전하거나 직원이 우회하면 자동화는 쓰이지 않습니다. 구축비뿐 아니라 요구사항 변경, 장애 대응, 교육, API 비용, 지원시간을 원가로 계산해야 합니다.

## 성패를 가르는 것

### 1. 자동화 전에 기준 업무를 측정해야 합니다
- 관찰: 고객은 “AI로 자동화해 달라”고 요청하지만 현재 처리시간·오류·승인 단계가 정리되지 않은 경우가 많습니다.
- 숨은 변수: 월 처리 건수, 건당 시간, 오류·재작업률, 승인자 수, 병목 담당자, 오류 한 건의 비용입니다.
- 작동 메커니즘: 자동화 전 기준값이 있어야 절감효과와 가격을 설명할 수 있습니다.
- 실행 판단: 실제 작업 20~50건을 관찰해 `입력 → 판단 → 승인 → 출력 → 보관 → 예외 → 재작업`을 기록합니다.
- 실패·전환 신호: 기준 시간과 오류를 측정할 수 없거나 담당자마다 방식이 다르면 개발보다 유료 진단으로 전환합니다.

### 2. 정상 흐름보다 예외 처리가 품질을 결정합니다
- 관찰: 데모는 정상 데이터로 작동하지만 현장에는 누락·중복·오타·권한 부족·긴급 처리·외부 장애가 있습니다.
- 숨은 변수: 예외율, 수동 전환 기준, 재실행, 중복 방지, 오류 로그, 알림, 복구시간입니다.
- 작동 메커니즘: 예외를 사람이 안전하게 처리할 수 있어야 자동화가 업무를 멈추지 않습니다.
- 실행 판단: 예외 목록, 수동 처리, 재실행, 상태 로그, 장애 알림, 책임자를 설계합니다.
- 실패·전환 신호: 실패 원인을 찾지 못하거나 수동 복구가 원래 업무보다 오래 걸리면 자동화 범위를 줄입니다.

### 3. AI 판단과 사람 승인을 오류비용에 따라 나눕니다
- 관찰: 분류·요약·초안은 빠르지만 계약·가격·인사·민원·상담 판단은 오류 비용이 큽니다.
- 숨은 변수: 오답 비용, 근거 추적, 승인 지연, 책임 소재, 민감정보, 전면 재검수 비율입니다.
- 작동 메커니즘: 낮은 위험은 자동 처리하고 높은 위험은 근거와 함께 사람에게 넘기면 안정성이 높아집니다.
- 실행 판단: 업무를 자동 실행, 사람 승인, 자동화 금지의 세 등급으로 나눕니다.
- 실패·전환 신호: 사람이 결과를 전부 다시 확인하거나 수정한다면 AI 단계가 추가 업무가 된 것입니다.

### 4. 데이터·권한·개인정보가 기술보다 먼저입니다
- 관찰: 고객 파일·메일·주문·상담·직원 데이터에는 개인정보와 영업정보가 포함될 수 있습니다.
- 숨은 변수: 수집 목적, 보관기간, 외부 API 전송, 접근권한, 관리자 로그, 계정 회수, 백업·삭제입니다.
- 작동 메커니즘: 데이터 흐름과 책임을 설명하지 못하면 작은 자동화도 보안·계약 위험이 됩니다.
- 실행 판단: `수집 → 전송 → 처리 → 저장 → 조회 → 백업 → 삭제`를 문서화하고 최소 권한을 적용합니다.
- 실패·전환 신호: 고객이 데이터 위치·접근자·삭제방법을 설명하지 못하면 연결을 중단합니다.

### 5. 구축비보다 변경·유지보수 계약이 사업성을 좌우합니다
- 관찰: 고객 업무·API·화면·조직이 바뀌면 자동화도 수정해야 합니다.
- 숨은 변수: 무상 하자, 기능 추가, 고객 업무 변경, 외부 서비스 변경, 월 점검, 긴급 장애, 인수인계입니다.
- 작동 메커니즘: 변경 범위를 계약하지 않으면 구축 뒤 무상 지원이 계속됩니다.
- 실행 판단: 하자·변경·운영지원·긴급대응·종료·인수인계를 구분하고 월 지원시간 상한을 둡니다.
- 실패·전환 신호: 유지보수 시간이 월 계약금보다 크거나 고객별 구조가 지나치게 달라지면 표준 패키지를 줄입니다.

### 6. 사례 콘텐츠는 기술 화면보다 전후 비용을 보여줘야 합니다
- 관찰: 화려한 화면은 관심을 만들지만 구매자는 실제 절감시간·오류 감소·운영 책임을 알고 싶어 합니다.
- 숨은 변수: 공개 가능한 정보, 전후 숫자의 근거, 익명화, 채널별 첫 문장, 사례 제작시간, 유효 상담률입니다.
- 작동 메커니즘: “AI를 연결했다”보다 “월 300건 업무에서 어떤 단계와 오류를 줄였는가”를 보여주면 구매자가 자신의 업무와 연결하기 쉽습니다.
- 실행 판단: 유튜브는 전체 흐름, 인스타그램은 전후 화면, 틱톡은 한 작업의 변화, X는 핵심 수치, 페이스북은 업종·지역 사례로 재가공합니다.
- 실패·전환 신호: 조회는 늘지만 유효 상담이 없거나 사례 공개가 비밀유지·개인정보를 침해하면 중단합니다.

## 핵심 현실 인사이트
- 구체적 관찰: 자동화 성공은 데모가 아니라 현장 사용률·예외 복구·절감시간으로 결정됩니다.
- 남들이 놓치는 숨은 변수: 기준 업무 측정, 예외율, 전면 재검수, 권한, 유지보수 부채, 사례의 수치 근거입니다.
- 결과가 달라지는 작동 메커니즘: 정상·예외·승인·로그·복구를 함께 설계하면 업무가 멈추지 않습니다.
- 임계점·시간차·교환비용: `문의 → 진단 계약 → 착수금 → 설계·구축 → 시험운영 → 검수·인수 → 잔금 → 유지보수 → 원화 계좌 입금`
- 초보자가 먼저 측정하거나 결정할 것: 월 처리량, 건당 시간, 오류비용, 자동화 금지 범위, 유지보수 상한입니다.
- 계속할 조건: 고객 절감효과가 유지비보다 크고 공급자의 지원시간 차감 순이익이 남는 경우입니다.
- 중단·전환 신호: 현장 사용률이 낮거나 사람이 결과를 전부 재검수하거나 지원시간이 계약금을 넘는 경우입니다.
- 대표 사례·수치·반례: 5분 작업을 하루 2건 자동화하는 것보다 30초 작업을 하루 수천 건 줄이는 편이 가치가 클 수 있습니다.
- 반복 확인된 구조인지 단일 사례의 편집 판단인지: 기준 업무·예외·유지보수를 함께 보는 것은 사업성 판단이며 계약·보안 조건은 공식 확인이 필요합니다.

## 시장 현실
- 네이버 블로그·카페·브런치·유튜브·커뮤니티에서 먼저 확인한 동향: 주문·상담·견적·보고·문서 처리처럼 반복 업무를 줄이는 소형 자동화 수요가 많습니다.
- 반복된 패턴: 도구 연결 화면과 완전자동화를 강조하고 예외·보안·유지보수·교육을 과소평가하는 사례가 반복됩니다.
- 상충되는 의견: 맞춤 구축과 표준 패키지 중 어느 쪽이 유리한지는 고객 업무의 예외와 반복성에 따라 달라집니다.
- 작성자 이해관계 또는 광고 가능성: 자동화 도구·강의·템플릿 판매자는 구현 난도를 낮게 보일 유인이 있습니다.
- 이미 포화된 부분: 범용 챗봇, 단순 알림 연결, 근거 없는 자동응답 데모입니다.
- 아직 남은 기회: 한 업종의 주문·상담·견적·보고·분류를 예외와 승인까지 포함해 표준화한 패키지입니다.

- 실제 구매자·독자·시청자·광고주: 소상공인, 쇼핑몰 운영자, 교육기관, 병원·상담기관, 제조·물류·재단·기업 실무팀
- 발견·유입 경로: 네이버 검색·블로그, 유튜브·인스타그램·틱톡·X·페이스북 사례 콘텐츠, 당근, 크몽, 숨고, 위시켓, 소개
- 판매 또는 광고가 일어나는 플랫폼: 유료 진단·직접 계약, 크몽·숨고·위시켓, 당근 비즈니스 프로필·지역광고
- 결제수단: 플랫폼 대금보호, 계좌이체, 신용카드·간편결제, 세금계산서 조건의 결제
- 수익 지급자: 플랫폼 또는 고객
- 플랫폼 수수료·광고비·환불·CS 비용: 플랫폼 수수료, 네이버 검색광고, Meta 광고, 진단·개발·API·서버·교육·장애대응·외주·보안·세금 비용
- 확정 수익 조건: 진단·구축·검수·인수 조건을 통과하고 환불·분쟁·대금 조건이 확정된 경우
- 실제 입금 시점과 경로: 착수금·중도금·잔금과 월 유지보수료가 플랫폼 또는 원화 계좌로 입금
- 반복 가능한 구조인지: 공통 업무를 표준 패키지로 만들고 유지보수 시간이 통제되어야 합니다.
- 대상 범위 교정: 한글 업무·원화 결제·원화 정산과 실제 지원 범위를 기준으로 합니다.

- 개인정보·영업정보를 최소 수집합니다.

- 무단 계정 공유, 근거 없는 완전자동화 약속, 의사결정 책임 회피를 피합니다.
- 로그·백업·장애복구·인수인계 없이 운영하지 않습니다.
- 고객 사례 공개 전 비밀유지·개인정보·성과 수치 사용 동의를 확인합니다.

## 총괄이 반드시 확인할 것
1. 데모와 현장 사용률·절감효과를 구분했는가.
2. 정상 흐름·예외·승인·로그·복구를 다뤘는가.
3. 구축비와 유지보수·장애대응 원가를 분리했는가.
4. 고객가치와 공급자 순이익을 각각 계산했는가.
5. 다섯 영상 채널의 사례 원소스멀티유즈를 반영했는가.
6. 실제 결제·정산·세금·보안 경로를 구체화했는가.
7. 확인 전 수수료·보안·개인정보 조건을 단정하지 않았는가.
