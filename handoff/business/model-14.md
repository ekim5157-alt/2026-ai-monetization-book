## 핵심 결론

템플릿·프롬프트팩·워크북 판매는 파일을 많이 만들어 묶는 사업이 아닙니다. 실제 성패는 **구매자가 반복해서 하던 판단과 입력을 얼마나 줄이는가 × 설명 없이 첫 결과를 만들 수 있는가 × 자기 업무에 맞게 바꿀 수 있는가 × 문의·환불·개정비를 제하고도 이익이 남는가**에서 갈립니다.

핵심 공식은 다음과 같습니다.

`반복 업무 빈도 × 절약시간 × 첫 사용 성공률 × 수정 가능성 × 재사용률 × 비용 차감 순수익`

예쁜 화면, 프롬프트 개수, 워크북 분량은 결제 전 관심을 만들 수 있습니다. 그러나 구매 후 복사, 접근 권한, 데이터 입력, 수식 수정, 변수 교체에서 막히면 사용되지 않습니다. 이 모델에서 가장 중요한 숫자는 판매 건수가 아니라 `첫 결과까지 걸린 시간`, `7일·30일 재사용률`, `구매자당 문의시간`입니다.

## 성패를 가르는 것

### 1. 파일이 아니라 반복되는 결정 하나를 줄여야 합니다

템플릿의 가치는 빈칸을 채워 주는 데 있지 않습니다. 사용자가 같은 업무를 할 때마다 다시 정하던 항목을 미리 구조화하는 데 있습니다.

예를 들어 콘텐츠 캘린더라면 날짜표만 제공해서는 부족합니다. 주제 후보, 대상 독자, 제작 상태, 배포 채널, CTA, 성과 기록, 다음 수정 판단이 한 흐름으로 이어져야 합니다.

먼저 다음을 확인합니다.

- 이 업무를 한 달에 몇 번 반복하는가
- 기존에는 몇 분 또는 몇 시간이 걸리는가
- 템플릿을 쓰면 어떤 결정을 하지 않아도 되는가
- 반드시 입력해야 할 값은 몇 개인가
- 한 사람이 쓰는가, 팀이 함께 쓰는가
- 다음 프로젝트에도 복제할 수 있는가

한 번에 두 시간을 줄이더라도 일생에 한 번 쓰는 문서는 재사용과 후속 구매 가능성이 낮습니다. 반대로 매주 20분을 줄이는 도구는 반복 사용과 추천을 만들 수 있습니다.

### 2. 다운로드보다 첫 사용 성공이 제품의 임계점입니다

파일 전달은 제품 사용이 아닙니다. 구매자가 첫 결과를 만들지 못하면 환불, 부정 후기, 추가 문의, 재구매 중단으로 이어집니다.

첫 사용 성공을 위해 필요한 것은 다음과 같습니다.

- 실제 예시 데이터
- 반드시 바꿔야 하는 항목 표시
- 복사·권한 설정 순서
- 예상 결과 화면
- 잘못 입력했을 때 복구 방법
- 처음 5분에 할 행동
- 자주 발생하는 오류 안내

기능을 추가하기 전에 첫 결과까지 필요한 클릭 수와 설명량을 줄여야 합니다.

### 3. 수정 가능한 영역과 보호해야 할 핵심을 나눠야 합니다

구매자의 업종, 팀 규모, 업무 순서는 서로 다릅니다. 제작자의 방식에만 맞는 템플릿은 보기에는 좋아도 실제 업무에 들어가기 어렵습니다.

다음 네 층을 분리합니다.

1. 바꾸면 안 되는 핵심 구조
2. 구매자가 반드시 바꿔야 하는 변수
3. 필요할 때만 여는 선택 기능
4. 숙련자용 확장 영역

모든 것을 자유롭게 바꾸게 하면 오류가 늘고, 아무것도 바꿀 수 없게 하면 적용 범위가 좁아집니다. 수정 가능성은 기능 수가 아니라 **안전하게 바꿀 수 있는 범위가 명확한가**로 판단합니다.

### 4. 프롬프트팩은 문장 묶음이 아니라 작업 프로토콜이어야 합니다

프롬프트는 모델, 입력자료, 대화 맥락, 언어에 따라 결과가 달라집니다. 한 줄 문장을 많이 모아도 사용자가 언제 어떤 정보를 넣어야 하는지 모르면 결과가 불안정합니다.

유료 프롬프트팩은 최소한 다음을 포함해야 합니다.

- 사용하는 업무 단계
- 필요한 입력자료
- 반드시 바꿀 변수
- 출력 형식
- 결과가 나쁠 때 수정할 항목
- 사실·수치·권리 검수 단계
- 다른 모델에서 조정하는 방법
- 사용하면 안 되는 상황

판매해야 하는 것은 ‘프롬프트 300개’가 아니라 **한 업무를 입력·생성·검수·수정까지 끝내는 절차**입니다.

### 5. 워크북은 질문 수가 아니라 결정을 끝내야 합니다

질문이 많은 워크북은 풍부해 보이지만, 완료 후 무엇을 선택해야 하는지 남지 않으면 사용자는 다시 검색하게 됩니다.

좋은 워크북은 다음을 남깁니다.

- 최종 선택안
- 선택 근거
- 포기한 대안
- 실행 순서
- 측정할 숫자
- 중단 기준

사업 아이디어 워크북이라면 아이디어를 많이 적게 하는 것으로 끝내지 않습니다. 수요, 실행비용, 회수기간, 반복 가능성으로 비교해 최종 후보를 좁혀야 합니다.

### 6. 도구 의존성이 높을수록 판매량이 개정 부채가 됩니다

Notion, Google Sheets, Microsoft Excel, Canva, AI 모델, 자동화 도구는 기능과 화면이 바뀔 수 있습니다. 특정 메뉴 위치, 모델 출력, 플러그인에 강하게 의존한 상품은 변경 때마다 수정과 안내가 필요합니다.

개정 비용은 다음에서 발생합니다.

- 화면 캡처 교체
- 공유·복제 방식 변경
- 깨진 수식과 연결 수정
- 프롬프트 재검수
- 기존 구매자 재배포
- 버전별 문의 대응

변하지 않는 업무 원리는 본 상품에 두고, 자주 바뀌는 화면과 기능은 온라인 사용가이드로 분리하는 편이 낫습니다.

### 7. 단품 수보다 업무의 시작과 끝을 연결해야 합니다

제안서 템플릿 하나보다 다음 흐름을 연결한 상품이 더 큰 가치를 만들 수 있습니다.

`고객 질문지 → 범위 정의표 → 견적 계산표 → 제안서 → 수정요청 기록표 → 납품 체크리스트`

다만 파일 수를 늘리는 것만으로는 부족합니다. 앞 단계의 입력이 다음 단계로 이어지고, 같은 정보를 다시 적는 횟수가 줄어야 합니다.

## 핵심 현실 인사이트

- 구체적 관찰: 구매자는 파일을 결제하지만 만족은 복사 후 첫 결과를 만드는 순간에 결정됩니다.
- 남들이 놓치는 숨은 변수: 반복 빈도, 구매자의 도구 숙련도, 수정해야 할 변수 수, 첫 결과까지의 시간, 구매자당 문의시간입니다.
- 결과가 달라지는 작동 메커니즘: 입력 순서와 수정 범위를 제한하면 오류가 줄고, 첫 사용과 반복 사용이 늘며, 문의·환불 비용이 낮아집니다.
- 임계점·시간차·교환비용: 첫 결제는 빠를 수 있지만 첫 사용, 7일·30일 재사용, 환불, 개정비를 지나야 실제 사업성이 드러납니다.
- 초보자가 먼저 측정하거나 결정할 것: 사용 빈도, 절약시간, 첫 결과시간, 수정 가능 영역, 지원 범위입니다.
- 계속할 조건: 구매자가 설명 없이 첫 결과를 만들고 반복 사용하며, 문의·환불·개정비를 뺀 순수익이 남는 경우입니다.
- 중단·전환 신호: 판매는 되지만 첫 사용 실패가 많거나, 도구 변경 때마다 전체 상품을 다시 만들어야 하는 경우입니다.
- 대표 사례·수치·반례: 프롬프트 300개보다 특정 업무 하나를 입력·생성·검수까지 끝내는 5개 프롬프트가 더 높은 실사용 가치를 만들 수 있습니다.
- 반복 확인된 구조인지 단일 사례의 편집 판단인지: 첫 사용 성공·재사용·지원시간을 핵심 변수로 보는 것은 사업 구조에 대한 편집 판단이며, 플랫폼별 가격·환불·정산 수치는 다음 단계가 확인해야 합니다.

## 수익 전 임계점과 필요한 작업량

1. 매주 또는 매월 반복되는 업무를 선택합니다.
2. 사용 전후 절약시간을 실제로 측정합니다.
3. 초보자가 설명 없이 첫 결과를 만듭니다.
4. 자기 상황에 맞게 핵심 변수를 수정합니다.
5. 7일·30일 뒤 다시 사용합니다.
6. 환불과 문의 뒤에도 건당 순이익이 남습니다.
7. 도구 변경 뒤 개정시간을 감당할 수 있습니다.

고정 기간을 보장할 수는 없습니다. 다음 순서로 진행합니다.

`최소 기능 제작 → 예시 데이터 입력 → 초보자 3~5명 사용 테스트 → 첫 사용 실패 수정 → 반복 문의를 FAQ·영상으로 전환 → 7일·30일 재사용 확인 → 첫 도구 변경 뒤 개정시간 측정`

상품 수를 늘리기 전에 한 상품의 첫 사용과 재사용을 확인해야 합니다.

## 관심·다운로드·판매 건수가 돈이 되지 않는 이유

- 사용영상 조회수는 도구에 대한 관심일 수 있습니다.
- 무료 템플릿 다운로드는 무료 자료 수집 행동일 수 있습니다.
- 유료 파일 전달은 첫 사용 성공을 뜻하지 않습니다.
- 판매 건수에는 할인·지인·충동구매가 섞일 수 있습니다.
- 후기 수는 30일 재사용을 보장하지 않습니다.
- 환불 전 매출은 확정 수익이 아닙니다.
- 판매가에는 문의·개정시간이 반영되지 않을 수 있습니다.

## 숨은 병목 5개

1. **첫 사용 실패** — 복사·권한·입력 단계에서 막힙니다.
2. **수정 난이도** — 자기 업무에 맞게 바꾸지 못해 일회성 파일이 됩니다.
3. **도구·모델 의존성** — 화면과 출력 방식이 바뀌면 빠르게 낡습니다.
4. **무료 대체재** — 디자인과 파일 수만으로는 무료 자료를 이기기 어렵습니다.
5. **지원시간 누적** — 저가 상품이라도 권한·수식 문의가 반복되면 마진이 사라집니다.

| 항목 | 의미 | 초보자의 오판 |
|---|---|---|
| 월 사용 횟수 | 반복 가치 | 한 번 절약되는 시간만 봄 |
| 사용 전후 처리시간 | 실제 절약효과 | 보기 좋은 화면을 효용으로 착각 |
| 첫 사용 완료율 | 실제 사용성 | 다운로드를 사용으로 간주 |

| 후속 상품 전환율 | 문제의 연속성 | 단품 판매량만 평가 |
| 비용 차감 순수익 | 실제 사업성 | 판매가를 이익으로 계산 |

## 첫 수익과 실제 입금의 시간 차이

`콘텐츠 노출 → 판매페이지 방문 → 결제 → 파일 전달·복제 → 첫 사용 → 환불·분쟁 가능 기간 → 판매·결제 수수료 반영 → 정산 확정 → 원화 계좌 입금`

결제 완료가 제품 성공을 뜻하지 않습니다. 첫 사용과 환불기간을 통과한 뒤 문의·개정비를 빼고 남는 금액이 실제 사업성에 가깝습니다.

## 실패 사례

- 프롬프트 수는 많지만 사용 시점과 입력방법이 없는 경우
- 제작자에게는 쉽지만 구매자는 필터·수식·관계를 수정하지 못하는 경우
- 예쁜 화면만 있고 실제 업무의 시작과 끝이 연결되지 않는 경우
- 도구 변경 뒤 사용가이드와 화면이 맞지 않는 경우
- 낮은 가격으로 팔았지만 권한·복사·수식 문의로 이익이 사라진 경우
- 무료 템플릿의 색상과 표지만 바꿔 판매한 경우

## 중단·전환 기준

- 결제는 되지만 첫 사용 완료율이 낮으면 기능보다 온보딩을 고칩니다.
- 같은 문의가 반복되면 설명 영상·예시 데이터·오류 안내를 추가합니다.
- 7일·30일 재사용이 거의 없으면 반복 업무가 아니라 일회성 호기심을 해결한 것입니다.
- 개정시간이 같은 기간 판매이익을 넘으면 도구 의존 부분을 분리하거나 판매를 중단합니다.
- 맞춤 설정 요청이 많으면 설치·세팅·컨설팅 서비스의 입문 상품으로 전환합니다.
- 무료 대체재와 가격 경쟁만 남으면 특정 직무·상황·결과로 더 좁힙니다.

## 뻔한 조언과 수정안

| 뻔한 표현 | 수정안 |
|---|---|
| 템플릿을 많이 만들어라 | 반복 업무 하나를 처음부터 끝까지 줄이는 구조를 만들어라 |
| 프롬프트를 모아 팔아라 | 입력·출력·검수·실패 수정법이 포함된 작업 프로토콜을 팔아라 |
| 예쁘게 디자인하라 | 첫 결과까지 필요한 클릭과 설명을 줄여라 |
| 기능을 많이 넣어라 | 자주 쓰는 핵심 기능만 남기고 오류와 학습비용을 줄여라 |
| AI로 자동 생성하라 | 실제 입력값과 여러 모델에서 결과를 검수하고 실패 조건을 설명하라 |

## 시장 현실

- 네이버 블로그·카페·브런치스토리·유튜브·커뮤니티에서 먼저 확인할 동향: 실제 구매 유형, 가격 저항, 첫 사용 실패, 환불 이유, 개정 중단 사례
- 반복된 패턴: 프롬프트 개수 경쟁, 무료 템플릿 재포장, 설명 부족, 도구 변경 뒤 방치
- 상충되는 의견: 저가 단품 대량판매와 직무별 업무 패키지 중 문의·개정비를 뺀 순이익이 더 높은 구조
- 작성자 이해관계 또는 광고 가능성: 템플릿 판매자와 강사의 성공 사례는 자기 상품 판매 목적이 있을 수 있으므로 구매자 후기와 교차 확인
- 이미 포화된 부분: 범용 일정표, 단순 프롬프트 모음, 질문만 많은 워크북
- 아직 남은 기회: 직무·업종별 실제 문서 흐름을 연결하고 첫 결과까지 안내하는 업무 완결형 패키지

## 판매·수익·광고 경로

- 실제 구매자·독자·시청자·광고주: 취업 준비자, 프리랜서, 소상공인, 콘텐츠 제작자, 팀 업무를 표준화하려는 실무자
- 발견·유입 경로: 네이버 검색·블로그·카페, 브런치스토리, 유튜브 사용영상, 인스타그램, 스레드, 뉴스레터, 기존 고객
- 판매 또는 광고가 일어나는 플랫폼: 크몽 등 디지털 상품 판매처, 자체 결제페이지, Notion Marketplace의 제출·판매 경로
- 결제수단: 신용카드, 네이버페이, 카카오페이, 토스페이, 계좌이체
- 수익 지급자: 판매 플랫폼, 템플릿 마켓 또는 결제대행사
- 플랫폼 수수료·광고비·환불·CS 비용: 판매·결제 수수료, 광고비, 환불, 구매자당 문의시간, 개정시간을 합산
- 확정 수익 조건: 환불·분쟁·정산 조정을 통과한 판매대금
- 실제 입금 시점과 경로: 플랫폼 정산주기와 최소 지급액을 통과한 뒤 원화 계좌 입금
- 반복 가능한 구조인지: 첫 사용과 30일 재사용이 확인되고 비용 차감 순수익이 남아야 함
- 해외 타깃과 구분할 부분: 다른 통화권 마켓은 언어, 업무 관행, 결제, 세금, 지원시간을 별도 계산

## 공식 확인 사실

사업성 분석 단계에서는 가격, 무료 한도, 템플릿 재판매 조건, 수수료, 환불기간, 지급일을 확정하지 않습니다. 다음 단계가 공식 문서로 확인해야 합니다.

- Notion의 템플릿 제출·복제·판매 조건
- Google Sheets·Microsoft Excel 파일의 배포·매크로·애드인 조건
- Canva 등 디자인 도구 내부 소재의 템플릿 재판매 조건
- AI 출력물과 프롬프트의 상업 이용·재판매 조건
- 크몽 등 판매처의 판매·결제 수수료와 정산 일정
- 공유 링크·복제·좌석·워크스페이스 제한
- 구매자 개인정보와 마케팅 동의 조건

## 현실 기준 교정

- 미국 기준 자료인지: 영어권 템플릿 마켓과 프롬프트 판매 사례는 별도 사례로 봅니다.
- 거주자·사업자 적용 가능 여부: 대한민국 거주 개인·개인사업자의 판매자 가입과 정산 가능 여부를 확인합니다.
- 결제·지급·정산 조건: 신용카드·간편결제·계좌이체와 원화 계좌 입금 조건을 확인합니다.
- 원화 가격·세금·환율·수수료: 종합소득세·부가가치세·판매수수료를 포함해 계산합니다.
- 언어 지원·지역 제한: 한글 템플릿의 검색 노출과 고객지원 가능성을 확인합니다.
- 플랫폼·소비 습관 차이: 영어권의 Notion·prompt 구매 습관을 원화 결제 사용자에게 그대로 적용하지 않습니다.
- 적용 확인이 필요한 항목: 실제 구매 가격대, 자주 쓰는 업무 도구, 구매 후 지원 기대, 간편결제 선호

## 추정 또는 편집 판단

- 첫 결과까지 10분 안팎을 목표로 하는 것은 절대 기준이 아니라 사용성 개선을 위한 편집 기준입니다.
- 첫 사용 완료율·30일 재사용률·구매자당 문의시간을 핵심 지표로 보는 것은 사업 구조에 대한 판단입니다.
- 단품보다 업무 흐름형 묶음이 유리할 수 있다는 것은 검증이 필요한 사업 가설입니다.
- 구체적인 가격대·전환율·환불률은 공식 자료와 현장 사례 확인 전이므로 단정하지 않습니다.

## 오늘·3일·7일·30일

### 오늘
- 한 달에 반복되는 업무 하나를 고릅니다.
- 사용 전후 처리시간과 반복 결정 항목을 기록합니다.
- 핵심 구조, 수정 변수, 보호 영역을 나눕니다.

### 3일
- 최소 기능 템플릿과 실제 예시 데이터를 만듭니다.
- 초보자용 기본 경로와 숙련자용 확장 경로를 분리합니다.
- 프롬프트라면 입력·출력·검수·실패 수정법을 함께 작성합니다.

### 7일
- 초보자 3~5명에게 설명 없이 사용하게 합니다.
- 첫 결과시간, 막힌 지점, 질문, 수정 횟수를 기록합니다.
- 다운로드가 아니라 첫 사용 완료를 측정합니다.

### 30일
- 결제율, 첫 사용 완료율, 7일·30일 재사용률, 문의, 환불, 순수익을 비교합니다.
- 반복 문의는 사용영상과 오류 안내로 전환합니다.
- 개정 부담이 큰 기능은 온라인 가이드나 별도 모듈로 분리합니다.

## 위험과 주의사항

- 타인의 템플릿을 일부 수정해 재판매하지 않습니다.
- 폰트·아이콘·플러그인·스크립트·예시 데이터의 배포 권한을 확인합니다.
- 프롬프트 결과를 보장하거나 AI 출력이 항상 동일하다고 표현하지 않습니다.
- 예시 데이터에 실제 고객·회사·개인의 민감정보를 넣지 않습니다.
- 공유 링크가 원본 편집 권한이나 다른 구매자의 데이터를 노출하지 않는지 확인합니다.
- 개정 지원기간과 무상 수정 범위를 명확히 합니다.

## 총괄이 반드시 확인할 것

1. 파일 수가 아니라 반복 업무의 시간 절감을 설명했는가.
2. 다운로드와 첫 사용·30일 재사용을 구분했는가.
3. 프롬프트를 문장 모음이 아니라 작업 프로토콜로 설명했는가.
4. 수정 가능 영역과 보호할 핵심을 구분했는가.
5. 문의·환불·개정시간을 원가에 포함했는가.
6. 실제 구매자·결제수단·지급자·원화 입금 경로가 구체적인가.
7. 확인 전 가격·수수료·환불기간을 사실처럼 단정하지 않았는가.
