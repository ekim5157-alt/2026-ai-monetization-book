## 핵심 결론

템플릿·프롬프트팩·워크북 판매는 파일을 많이 만들어 묶는 사업이 아니라 **반복 업무에서 사용자가 매번 다시 하던 판단을 줄여 주는 사업**입니다.

성패는 다음 공식에서 갈립니다.

`반복 빈도가 높은 업무 × 첫 사용 성공 × 수정 가능한 구조 × 재사용 가치 × 지원·업데이트를 뺀 순이익`

예쁜 화면, 프롬프트 개수, 워크북 분량은 구매 전 관심을 만들 수 있습니다. 그러나 구매 후 복사·권한 설정·입력·수식 수정·프롬프트 변수 변경에서 막히면 사용되지 않습니다. 따라서 판매량보다 중요한 숫자는 첫 사용 완료율, 첫 결과까지 걸린 시간, 7일·30일 재사용률, 구매자당 문의시간입니다.

좋은 상품은 사용자가 템플릿을 이해하게 만드는 것이 아니라, **사용자가 자기 업무를 더 빨리 끝내게 만드는 것**입니다.

## 성패를 가르는 것

### 1. 사용 빈도가 낮은 업무는 시간 절감이 커도 반복 매출을 만들기 어렵습니다

템플릿의 가치는 한 번 사용할 때 절약되는 시간과 얼마나 자주 사용하는지를 함께 봐야 합니다.

- 한 달에 한 번 쓰는 자료인지
- 매일·매주 반복되는 업무인지
- 한 번 쓰고 버리는 제출 양식인지
- 여러 프로젝트에서 복제하는 운영 도구인지
- 한 사람이 쓰는지 팀이 함께 쓰는지

한 번에 두 시간을 줄이더라도 일생에 한 번 쓰는 문서는 재사용과 후속 상품 가능성이 낮습니다. 반대로 매주 20분을 줄이는 템플릿은 반복 사용과 추천 가능성이 높을 수 있습니다.

초보자는 ‘멋진 결과 화면’을 먼저 만들기 쉽지만, 사업자는 `사용 빈도 × 절약시간`을 먼저 계산해야 합니다.

### 2. 다운로드가 아니라 첫 사용 성공이 제품의 임계점입니다

디지털 파일은 전달되면 판매가 끝난 것처럼 보입니다. 그러나 구매자가 첫 결과를 만들지 못하면 환불, 부정 후기, 추가 문의, 재구매 중단으로 이어집니다.

첫 사용 성공을 위해 필요한 것은 다음과 같습니다.

- 실제 예시 데이터
- 반드시 바꿔야 하는 항목 표시
- 복사·권한 설정 절차
- 예상 결과 화면
- 잘못 입력했을 때 복구법
- 초보자용 짧은 사용 안내
- 자주 발생하는 오류 목록

첫 사용까지 설명이 길어질수록 저가 상품의 지원비가 커집니다. 기능을 추가하기 전에 첫 결과까지의 클릭 수와 설명량을 줄여야 합니다.

### 3. 수정 가능한 구조가 아니면 일회성 샘플에 불과합니다

구매자의 업종, 업무 순서, 데이터 항목은 서로 다릅니다. 제작자의 방식에만 맞는 템플릿은 샘플로는 좋아 보여도 실제 업무에 들어가기 어렵습니다.

수정 가능한 상품은 다음을 구분합니다.

- 반드시 유지해야 하는 핵심 구조
- 사용자가 바꿔야 하는 변수
- 초보자가 건드리면 안 되는 수식·연결
- 업종별로 추가할 선택 항목
- 기본형과 고급형의 차이

모든 기능을 자유롭게 바꾸게 하면 오류가 늘고, 아무것도 바꿀 수 없게 하면 적용 범위가 좁아집니다. **변경 가능한 부분과 고정해야 할 부분을 명확히 나누는 설계 감각**이 결과를 가릅니다.

### 4. 프롬프트팩은 문장 모음이 아니라 작업 프로토콜이어야 합니다

프롬프트는 모델, 언어, 입력자료, 대화 맥락에 따라 결과가 달라집니다. 한 줄 문장을 많이 모아도 사용자가 언제 어떤 정보를 넣어야 하는지 모르면 결과가 불안정합니다.

유료 프롬프트팩은 최소한 다음을 포함해야 합니다.

- 사용 시점
- 필요한 입력자료
- 반드시 바꿀 변수
- 원하는 출력 형식
- 결과가 나쁠 때 수정할 항목
- 사실·수치·권리를 검수하는 단계
- 다른 모델에서 조정하는 방법

판매해야 하는 것은 ‘프롬프트 300개’가 아니라 **한 업무를 입력·생성·검수·수정까지 끝내는 절차**입니다.

### 5. 워크북은 질문 수가 아니라 마지막 결정이 명확해야 합니다

질문을 많이 던지는 워크북은 풍부해 보이지만, 완료 후 무엇을 선택해야 하는지 정리되지 않으면 사용자는 다시 검색하게 됩니다.

좋은 워크북은 다음을 남깁니다.

- 최종 선택안
- 선택 근거
- 포기한 대안
- 실행 순서
- 측정할 숫자
- 중단 기준

사업 아이디어 워크북이라면 아이디어를 50개 적게 하는 것이 아니라 수요, 비용, 예상 수익, 회수기간, 반복 가능성으로 비교해 최종 후보를 좁혀야 합니다.

### 6. 도구와 모델에 강하게 묶일수록 판매량이 업데이트 부채가 됩니다

노션, 스프레드시트, 디자인 도구, AI 모델, 자동화 도구는 기능과 화면이 바뀔 수 있습니다. 특정 메뉴 위치, 특정 모델의 출력, 특정 플러그인에 의존한 상품은 변경 때마다 수정과 안내가 필요합니다.

업데이트 비용은 다음에서 발생합니다.

- 화면 캡처 교체
- 권한·복제 방식 변경
- 깨진 수식과 자동화 수정
- 프롬프트 재검수
- 기존 구매자 재배포
- 버전별 문의 대응

핵심 구조와 도구 사용가이드를 분리해야 합니다. 변하지 않는 업무 원리는 본 상품에 두고, 자주 바뀌는 화면과 기능은 온라인 가이드로 관리하는 편이 낫습니다.

### 7. 단품보다 업무의 시작과 끝을 연결한 묶음이 강할 수 있습니다

단일 템플릿은 무료 대체재와 직접 비교됩니다. 반면 한 업무를 처음부터 끝까지 연결하는 상품은 구매 이유가 명확해질 수 있습니다.

예를 들어 제안서 템플릿 하나보다 다음 흐름을 연결할 수 있습니다.

- 고객 질문지
- 범위 정의표
- 견적 계산표
- 제안서
- 수정요청 기록표
- 납품 체크리스트

그러나 파일 수를 늘리는 것만으로는 가치가 생기지 않습니다. 각 파일이 다음 단계로 데이터를 넘기고 사용자의 재입력을 줄여야 합니다.

### 숨은 병목 5개

1. **첫 사용 실패** — 복사·권한·입력 단계에서 막혀 환불과 문의가 생깁니다.
2. **수정 난이도** — 자기 업무에 맞게 바꾸지 못하면 일회성 파일이 됩니다.
3. **도구·모델 의존성** — 화면과 출력 방식이 바뀌면 상품이 빠르게 낡습니다.
4. **무료 대체재** — 디자인과 파일 수만으로는 무료 자료를 이기기 어렵습니다.
5. **지원시간 누적** — 저가 상품이라도 설치·권한·수식 문의가 반복되면 마진이 사라집니다.

### 초보자가 잘못 읽기 쉬운 숫자

| 잘못 읽는 숫자 | 왜 위험한가 | 대신 볼 숫자 |
|---|---|---|
| 파일 수 | 업무 완료와 비례하지 않음 | 하나의 업무에서 줄어든 단계 수 |
| 다운로드 수 | 실제 사용 여부를 모름 | 첫 사용 완료율 |
| 기능 수 | 학습·오류 비용이 늘 수 있음 | 첫 결과까지 걸린 시간 |
| 판매 건수 | 환불·문의가 섞임 | 지원비 반영 확정 순수익 |
| 후기 수 | 할인·지인 후기가 섞일 수 있음 | 비지인 재사용·재구매율 |
| 프롬프트 수 | 출력 안정성과 무관 | 업무별 성공률·수정 횟수 |

### 실패 사례

- 프롬프트 300개를 묶었지만 사용 시점과 입력방법이 없는 경우
- 제작자에게는 쉽지만 구매자는 필터·수식·관계를 수정하지 못하는 경우
- 예쁜 화면만 있고 실제 업무의 시작과 끝이 연결되지 않는 경우
- 도구 업데이트 후 사용가이드와 화면이 맞지 않는 경우
- 낮은 가격으로 팔았지만 권한·복사·수식 문의로 건당 이익이 사라진 경우
- 무료 템플릿을 색상과 표지만 바꿔 판매한 경우

### 중단·전환 기준

- 결제는 되지만 첫 사용 완료율이 낮으면 기능 추가보다 온보딩을 고칩니다.
- 같은 문의가 반복되면 설명 영상, 예시 데이터, 오류 안내를 추가합니다.
- 7일·30일 재사용이 거의 없으면 반복 업무가 아니라 일회성 호기심을 해결한 것입니다.
- 업데이트시간이 같은 기간 판매이익을 넘으면 도구 의존 부분을 분리하거나 판매를 중단합니다.
- 맞춤 설정 요청이 많으면 단품을 설치·세팅·컨설팅 서비스의 입문 상품으로 전환합니다.
- 무료 대체재와 가격 경쟁만 남으면 특정 직무·상황·결과로 더 좁힙니다.

### 뻔한 조언과 수정안

| 뻔한 표현 | 수정안 |
|---|---|
| 템플릿을 많이 만들어라 | 반복 업무 하나를 처음부터 끝까지 줄이는 구조를 만들어라 |
| 프롬프트를 모아 팔아라 | 입력값·출력형식·검수·실패 수정법이 포함된 작업 프로토콜을 팔아라 |
| 예쁘게 디자인하라 | 구매 후 첫 결과까지의 클릭과 설명을 줄여라 |
| 기능을 많이 넣어라 | 자주 쓰는 핵심 기능만 남기고 오류와 학습비용을 줄여라 |
| AI로 자동 생성하라 | 실제 입력값과 여러 모델에서 결과를 검수하고 실패 조건을 설명하라 |

## 핵심 현실 인사이트
- 구체적 관찰: 구매자는 파일을 사지만 만족은 복사 후 첫 결과를 만드는 순간에 결정됩니다.
- 남들이 놓치는 숨은 변수: 사용 빈도, 구매자의 도구 숙련도, 수정 변수의 수, 첫 결과까지의 클릭 수, 지원시간입니다.
- 결과가 달라지는 작동 메커니즘: 입력값과 사용 순서를 제한하면 초보자의 실패가 줄고 반복 사용과 재구매 가능성이 높아집니다.
- 임계점·시간차·교환비용: 첫 결제는 빠를 수 있지만 첫 사용·재사용·환불·지원·업데이트를 지나야 실제 사업성이 드러납니다.
- 초보자가 먼저 측정하거나 결정할 것: 사용 빈도, 절약시간, 첫 결과까지 걸린 시간, 수정 가능한 범위, 지원 범위입니다.
- 계속할 조건: 구매자가 설명 없이 첫 결과를 만들고 반복 사용하며 지원·업데이트비를 뺀 순수익이 남는 경우입니다.
- 중단·전환 신호: 판매는 되지만 첫 사용 실패와 문의가 많거나 도구 업데이트 때마다 전체 상품을 다시 만들어야 하는 경우입니다.
- 대표 사례·수치·반례: 프롬프트 300개보다 특정 업무 하나를 입력·생성·검수까지 끝내는 5개 프롬프트가 더 높은 실사용 가치를 만들 수 있습니다.
- 반복 확인된 구조인지 단일 사례의 편집 판단인지: 첫 사용 성공·재사용·지원시간을 핵심 변수로 보는 것은 사업 구조에 대한 편집 판단이며 국내 판매·환불 데이터는 조사 에이전트가 확인해야 합니다.

## 시장 현실
- 네이버 블로그·카페·브런치·유튜브·커뮤니티에서 먼저 확인한 동향: 사업성 분석 단계에서는 현장 자료를 직접 검증하지 않았습니다. 국내 템플릿 판매자·구매자 후기, 프롬프트팩 사용 후기, 노션·스프레드시트 사용 실패 사례를 우선 조사해야 합니다.
- 반복된 패턴: 프롬프트 개수 경쟁, 무료 템플릿의 재포장, 사용 설명 부족, 도구 업데이트 후 방치가 반복되는지 확인해야 합니다.
- 상충되는 의견: 저가 단품 대량판매와 직무별 업무 패키지 판매 중 어느 쪽이 지원비를 뺀 순이익이 높은지 비교해야 합니다.
- 작성자 이해관계 또는 광고 가능성: 템플릿 판매자와 강사의 성공 사례는 자신의 상품 판매 목적이 있을 수 있으므로 구매자 후기와 교차 확인해야 합니다.
- 이미 포화된 부분: 범용 노션 템플릿, 단순 프롬프트 모음, 질문만 많은 워크북이라는 가설을 검증해야 합니다.
- 아직 남은 기회: 한국의 직무·문서·고객응대 흐름에 맞춘 업무 완결형 패키지라는 가설을 검증해야 합니다.

## 판매·수익·광고 경로

- 결제수단: 국내 카드·간편결제·계좌 결제 제공 여부 확인
- 수익 지급자: 판매 플랫폼, 템플릿 마켓 또는 결제대행사
- 플랫폼 수수료·광고비·환불·CS 비용: 공식 수수료와 구매자당 지원시간·업데이트시간을 합산

- 실제 입금 시점과 경로: 플랫폼 정산주기, 최소 지급액, 국내 계좌 지급 조건 확인

## 공식 확인 사실

사업성 분석 단계에서 가격, 무료 한도, 템플릿 재판매 조건, 수수료, 지급일을 공식 사실로 확정하지 않았습니다. 다음 항목은 조사 에이전트가 공식 문서로 확인해야 합니다.

- 노션·스프레드시트·디자인 도구의 템플릿 재판매 조건
- 공유 링크·복제·좌석·워크스페이스 제한
- AI 출력물과 프롬프트의 상업 이용·재판매 조건
- 폰트·아이콘·이미지·플러그인·예시 데이터의 재배포 조건
- 판매 플랫폼 수수료·정산·환불 조건
- 파일 크기·다운로드·업데이트 전달 한도
- 구매자 개인정보와 이메일 사용 조건

## 현실 기준 교정
- 미국 기준 자료인지: 해외 템플릿 마켓과 프롬프트 판매 사례는 해외 기준으로 구분합니다.
- 거주자·사업자 적용 가능 여부: 한국 거주 개인·사업자의 판매자 가입과 정산 가능 여부를 확인합니다.
- 결제·지급·정산 조건: 국내 계좌 지급, 해외 지급수단, 최소 지급액과 보류 조건을 구분합니다.
- 원화 가격·세금·환율·수수료: 해외 판매는 환율·환전·해외결제·세금을 포함해 계산합니다.
- 언어 지원·지역 제한: 한국어 템플릿과 프롬프트가 해당 마켓에서 발견되고 지원 가능한지 확인합니다.
- 플랫폼·소비 습관 차이: 영어권의 Notion·prompt 상품 구매 습관을 국내에 그대로 적용하지 않습니다.
- 적용 확인이 필요한 항목: 국내 구매 가격대, 실제 사용 도구, 국내 업무 문서 형식, 구매 후 지원 기대

- 도구 업데이트 지원기간과 무상 수정 범위를 명확히 해야 합니다.

4. 구매자의 숙련도 차이와 수정 난이도를 핵심 변수로 다뤘는가.
5. 지원시간과 업데이트시간을 원가에 포함했는가.
6. 무료 대체재와의 차이를 디자인이 아니라 결과와 사용성으로 설명했는가.
7. 시장 자료 미확인 부분을 사실처럼 단정하지 않았는가.
