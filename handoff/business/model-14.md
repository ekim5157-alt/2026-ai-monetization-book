## 핵심 결론

템플릿·프롬프트팩·워크북 판매는 파일을 많이 만드는 사업이 아니라 **사용자가 반복해서 하던 판단과 입력을 줄여 업무를 더 빨리 끝내게 하는 사업**입니다.

성패 공식은 다음과 같습니다.

`반복 빈도가 높은 업무 × 첫 사용 성공 × 수정 가능한 구조 × 재사용 가치 × 지원·업데이트를 뺀 순이익`

예쁜 화면, 프롬프트 개수, 워크북 분량은 구매 전 관심을 만들 수 있습니다. 그러나 구매 후 복사·권한 설정·입력·수식 수정·프롬프트 변수 변경에서 막히면 사용되지 않습니다. 따라서 판매량보다 첫 사용 완료율, 첫 결과까지 걸린 시간, 7일·30일 재사용률, 구매자당 문의시간이 중요합니다.

## 성패를 가르는 것

### 1. 사용 빈도가 높은 업무를 골라야 합니다

템플릿 가치는 한 번 사용할 때 절약되는 시간과 사용 빈도를 함께 봐야 합니다. 일생에 한 번 쓰는 문서보다 매주 반복하는 업무를 줄이는 도구가 재사용과 후속 구매를 만들기 쉽습니다.

먼저 다음을 확인합니다.

- 한 달에 몇 번 반복하는가
- 기존에는 몇 분·몇 시간이 걸리는가
- 매번 다시 결정하는 항목은 무엇인가
- 한 사람이 쓰는가, 팀이 함께 쓰는가
- 다른 프로젝트에도 복제할 수 있는가

### 2. 다운로드가 아니라 첫 사용 성공이 임계점입니다

파일 전달은 판매 완료가 아닙니다. 구매자가 첫 결과를 만들지 못하면 환불, 문의, 부정 후기, 재구매 중단으로 이어집니다.

첫 사용 성공을 위해 다음이 필요합니다.

- 실제 예시 데이터
- 반드시 바꿀 항목 표시
- 복사·권한 설정 절차
- 예상 결과 화면
- 잘못 입력했을 때 복구법
- 초보자용 짧은 안내

기능을 추가하기 전에 첫 결과까지의 클릭 수와 설명량을 줄여야 합니다.

### 3. 수정 가능한 부분과 고정할 부분을 나눠야 합니다

구매자의 업종과 업무 순서는 서로 다릅니다. 제작자의 방식에만 맞는 템플릿은 샘플로는 좋아 보여도 실제 업무에 들어가기 어렵습니다.

- 유지해야 하는 핵심 구조
- 사용자가 바꿔야 하는 변수
- 초보자가 건드리면 안 되는 수식·연결
- 업종별 선택 항목
- 기본형과 고급형의 차이

모든 것을 자유롭게 바꾸게 하면 오류가 늘고, 아무것도 바꿀 수 없게 하면 적용 범위가 좁아집니다.

### 4. 프롬프트팩은 문장 모음이 아니라 작업 프로토콜이어야 합니다

프롬프트는 모델, 언어, 입력자료, 대화 맥락에 따라 결과가 달라집니다. 유료 프롬프트팩은 다음을 포함해야 합니다.

- 사용 시점
- 필요한 입력자료
- 반드시 바꿀 변수
- 출력 형식
- 결과가 나쁠 때 수정할 항목
- 사실·수치·권리 검수 단계
- 다른 모델에서 조정하는 방법

판매해야 하는 것은 ‘프롬프트 300개’가 아니라 **한 업무를 입력·생성·검수·수정까지 끝내는 절차**입니다.

### 5. 워크북은 질문 수보다 마지막 결정이 중요합니다

좋은 워크북은 생각을 늘리는 것이 아니라 다음을 남겨야 합니다.

- 최종 선택안
- 선택 근거
- 포기한 대안
- 실행 순서
- 측정할 숫자
- 중단 기준

질문이 많아도 마지막 결정이 남지 않으면 사용자는 다시 검색하게 됩니다.

### 6. 도구와 모델 의존성이 업데이트 부채를 만듭니다

노션, 스프레드시트, 디자인 도구, AI 모델, 자동화 도구는 기능과 화면이 바뀔 수 있습니다.

업데이트 비용은 다음에서 발생합니다.

- 화면 캡처 교체
- 권한·복제 방식 변경
- 깨진 수식·자동화 수정
- 프롬프트 재검수
- 기존 구매자 재배포
- 버전별 문의 대응

변하지 않는 업무 구조와 자주 바뀌는 도구 사용가이드를 분리해야 합니다.

### 7. 단품보다 업무의 시작과 끝을 연결해야 합니다

제안서 템플릿 하나보다 고객 질문지, 범위 정의표, 견적 계산표, 제안서, 수정요청 기록표, 납품 체크리스트를 하나의 흐름으로 연결한 상품이 더 강할 수 있습니다.

다만 파일 수만 늘려서는 안 됩니다. 각 파일이 다음 단계로 정보를 넘기고 재입력을 줄여야 합니다.

## 수익이 생기기 전 통과해야 할 임계점

1. 한 달에 반복되는 업무여야 합니다.
2. 사용 전후 절약시간이 측정돼야 합니다.
3. 초보자가 설명 없이 첫 결과를 만들어야 합니다.
4. 자기 상황에 맞게 핵심 변수를 수정할 수 있어야 합니다.
5. 7일·30일 뒤에도 다시 사용해야 합니다.
6. 환불과 문의 뒤에도 건당 순이익이 남아야 합니다.
7. 도구 업데이트 뒤 수정비를 감당할 수 있어야 합니다.

## 무엇을 얼마나 버텨야 하는가

초기에는 상품 수를 늘리는 것이 아니라 한 상품의 첫 사용 실패를 반복해서 고쳐야 합니다.

- 초보자 테스트에서 막히는 지점 수정
- 예시 데이터와 안내 문구 보완
- 같은 문의를 FAQ·영상으로 전환
- 실제 업무에서 7일·30일 재사용 확인
- 도구 업데이트 뒤 수정시간 측정

고정 기간을 보장할 수는 없습니다. 다만 재사용이 확인되기 전에 상품 수를 늘리면 지원과 업데이트 부채만 커질 수 있습니다.

## 왜 조회수·다운로드·판매 건수가 돈이 되지 않는가

- 사용영상 조회수는 도구에 대한 관심일 수 있습니다.
- 무료 템플릿 다운로드는 무료 수집 행동일 수 있습니다.
- 유료 파일 다운로드는 첫 사용 성공을 뜻하지 않습니다.
- 판매 건수에는 할인·지인·충동구매가 섞일 수 있습니다.
- 후기 수는 실제 재사용과 무관할 수 있습니다.
- 환불 전 매출은 확정 수익이 아닙니다.

따라서 첫 사용 완료율, 재사용률, 문의시간, 환불률, 건당 순수익을 봐야 합니다.

## 같은 도구를 써도 결과가 달라지는 운영 판단

- 반복 빈도가 높은 업무를 고르는 능력
- 핵심 구조와 수정 변수를 분리하는 능력
- 초보자가 건드리면 안 되는 부분을 보호하는 능력
- 프롬프트를 문장이 아니라 입력·출력·검수 절차로 만드는 능력
- 기능을 늘리기보다 첫 결과까지의 단계를 줄이는 능력
- 변하지 않는 업무 원리와 도구별 화면 안내를 분리하는 능력

## 숨은 병목 5개

1. 첫 사용 실패
2. 자기 업무에 맞게 수정하지 못함
3. 도구·모델 변경으로 빠르게 낡음
4. 무료 대체재와 차이가 없음
5. 설치·권한·수식 문의로 지원비가 누적됨

## 숫자로 측정할 항목

| 항목 | 의미 | 초보자의 오판 |
|---|---|---|
| 사용 빈도 | 반복 가치 | 한 번 절약되는 시간만 봄 |
| 첫 사용 완료율 | 실제 사용성 | 다운로드를 사용으로 간주 |
| 첫 결과까지 걸린 시간 | 온보딩 난이도 | 기능 수가 많을수록 좋다고 판단 |

| 구매자당 문의시간 | 숨은 원가 | 디지털 상품이라 CS가 없다고 가정 |
| 환불률 | 기대와 사용성 일치 | 결제를 확정 매출로 기록 |
| 업데이트당 수정시간 | 유지비 | 최초 제작시간만 계산 |
| 후속 상품 전환율 | 업무 문제의 연속성 | 단품 판매량만 평가 |
| 지원비 반영 순수익 | 실제 사업성 | 판매가를 이익으로 계산 |

## 첫 수익과 실제 입금의 시간 차이

`콘텐츠 노출 → 판매페이지 방문 → 결제 → 파일 전달·복제 → 첫 사용 → 환불·분쟁 가능 기간 → 정산 확정 → 실제 입금`

결제 완료는 제품 성공이 아닙니다. 첫 사용과 환불기간을 통과한 뒤 지원비를 빼고 남는 금액이 실제 수익에 가깝습니다.

## 실패 사례

- 프롬프트 300개를 묶었지만 사용 시점과 입력방법이 없는 경우
- 제작자에게는 쉽지만 구매자는 필터·수식·관계를 수정하지 못하는 경우
- 예쁜 화면만 있고 업무의 시작과 끝이 연결되지 않는 경우
- 도구 업데이트 후 사용가이드와 화면이 맞지 않는 경우
- 낮은 가격으로 팔았지만 권한·복사·수식 문의로 이익이 사라진 경우
- 무료 템플릿을 색상과 표지만 바꿔 판매한 경우

## 중단·전환 기준

- 결제는 되지만 첫 사용 완료율이 낮으면 기능보다 온보딩을 고칩니다.
- 같은 문의가 반복되면 설명 영상·예시 데이터·오류 안내를 추가합니다.
- 7일·30일 재사용이 거의 없으면 반복 업무가 아니라 일회성 호기심을 해결한 것입니다.
- 업데이트시간이 판매이익을 넘으면 도구 의존 부분을 분리하거나 판매를 중단합니다.
- 맞춤 설정 요청이 많으면 설치·세팅·컨설팅 서비스로 전환합니다.
- 무료 대체재와 가격 경쟁만 남으면 특정 직무·상황·결과로 좁힙니다.

## 뻔한 조언과 수정안

| 뻔한 표현 | 수정안 |
|---|---|
| 템플릿을 많이 만들어라 | 반복 업무 하나를 처음부터 끝까지 줄여라 |
| 프롬프트를 모아 팔아라 | 입력·출력·검수·실패 수정법이 포함된 프로토콜을 팔아라 |
| 예쁘게 디자인하라 | 첫 결과까지의 클릭과 설명을 줄여라 |
| 기능을 많이 넣어라 | 자주 쓰는 핵심 기능만 남겨라 |
| AI로 자동 생성하라 | 실제 입력값과 여러 모델에서 결과를 검수하라 |

## 핵심 현실 인사이트
- 구체적 관찰: 구매자는 파일을 사지만 만족은 첫 결과를 만드는 순간에 결정됩니다.
- 남들이 놓치는 숨은 변수: 사용 빈도, 수정 변수의 수, 첫 결과까지의 시간, 지원시간입니다.
- 결과가 달라지는 작동 메커니즘: 입력과 수정 범위를 제한하면 실패와 문의가 줄어듭니다.
- 임계점·시간차·교환비용: 첫 결제 뒤 첫 사용·재사용·환불·업데이트를 지나야 사업성이 드러납니다.
- 초보자가 먼저 측정할 것: 사용 빈도, 절약시간, 첫 결과시간, 수정 범위입니다.
- 계속할 조건: 설명 없이 첫 결과가 나오고 반복 사용되며 순수익이 남는 경우입니다.
- 중단 신호: 첫 사용 실패가 많거나 업데이트 때마다 전체 상품을 다시 만드는 경우입니다.
- 대표 반례: 프롬프트 300개보다 업무 하나를 끝내는 5개 프롬프트가 더 큰 가치를 만들 수 있습니다.
- 검증 수준: 사업 구조에 대한 편집 판단이며 국내 판매·환불 수치는 후속 조사가 필요합니다.

## 시장 현실
- 먼저 확인할 자료: 국내 템플릿 판매자·구매자 후기, 프롬프트팩 사용 후기, 노션·스프레드시트 사용 실패 사례
- 확인할 반복 패턴: 프롬프트 개수 경쟁, 무료 템플릿 재포장, 설명 부족, 업데이트 후 방치
- 상충 의견: 저가 단품 대량판매와 직무별 업무 패키지 중 실제 순이익이 높은 구조
- 이해관계: 판매자·강사의 성공 사례는 자기 상품 판매 목적이 있을 수 있어 구매자 후기와 교차 확인 필요
- 포화 가설: 범용 노션 템플릿, 단순 프롬프트 모음, 질문만 많은 워크북
- 기회 가설: 한국의 직무·문서·고객응대 흐름에 맞춘 업무 완결형 패키지

## 판매·수익·광고 경로
- 실제 구매자: 취업 준비자, 프리랜서, 소상공인, 콘텐츠 제작자, 업무를 표준화하려는 개인·팀
- 발견 경로: 네이버 검색·블로그·카페, 브런치, 국내 유튜브 사용영상, SNS, 뉴스레터
- 판매 경로: 국내 디지털 상품 판매 플랫폼, 자체 결제페이지, 도구별 템플릿 마켓
- 결제수단: 국내 카드·간편결제·계좌 결제 제공 여부 확인
- 수익 지급자: 판매 플랫폼, 템플릿 마켓 또는 결제대행사
- 비용: 공식 수수료와 지원·업데이트시간을 합산
- 확정 수익: 환불·분쟁·정산 조정을 통과한 판매대금
- 실제 입금: 정산주기·최소 지급액·국내 계좌 조건 확인
- 반복 구조: 첫 사용과 재사용이 확인되고 순수익이 남아야 함
- 해외 구분: 해외 마켓은 언어·업무 관행·결제·세금·지원시간을 별도 계산

## 공식 확인 사실

사업성 분석 단계에서는 가격, 무료 한도, 재판매 조건, 수수료, 지급일을 확정하지 않습니다. 조사 에이전트가 다음을 공식 문서로 확인해야 합니다.

- Notion·스프레드시트·디자인 도구의 템플릿 재판매 조건
- 공유 링크·복제·좌석·워크스페이스 제한
- AI 출력물과 프롬프트의 상업 이용·재판매 조건
- 폰트·아이콘·이미지·플러그인·예시 데이터의 재배포 조건
- 판매 플랫폼 수수료·정산·환불 조건
- 파일 크기·다운로드·업데이트 전달 한도
- 구매자 개인정보와 이메일 사용 조건

## 현실 기준 교정
- 해외 템플릿 마켓 사례는 국내 기준과 분리합니다.
- 한국 거주 개인·사업자의 판매자 가입·정산 가능 여부를 확인합니다.
- 국내 계좌 지급과 해외 지급수단을 구분합니다.
- 해외 판매는 환율·환전·세금·해외결제비를 포함합니다.
- 영어권의 도구 사용·구매 습관을 국내에 그대로 적용하지 않습니다.

## 추정 또는 편집 판단

- 첫 결과까지 10분 안팎을 목표로 하는 것은 절대 기준이 아니라 사용성 개선 기준입니다.
- 첫 사용 완료율·재사용률·지원시간을 핵심 지표로 보는 것은 사업 구조에 대한 판단입니다.
- 단품보다 업무 흐름형 묶음이 유리할 수 있다는 것은 검증이 필요한 가설입니다.
- 국내 가격대·전환율·환불률은 자료 확인 전이므로 단정하지 않습니다.

## 위험과 주의사항

- 타인의 템플릿을 일부 수정해 재판매하지 않습니다.
- 폰트·아이콘·플러그인·스크립트·예시 데이터의 배포 권한을 확인합니다.
- 프롬프트 결과를 보장하거나 출력이 항상 동일하다고 표현하지 않습니다.
- 예시 데이터에 실제 고객·회사의 민감정보를 넣지 않습니다.
- 공유 링크가 원본 편집 권한이나 다른 구매자의 데이터를 노출하지 않는지 확인합니다.
- 업데이트 지원기간과 무상 수정 범위를 명확히 합니다.

## 총괄이 반드시 확인할 것

1. 파일 수가 아니라 반복 업무의 시간 절감을 설명했는가.
2. 다운로드와 첫 사용·재사용을 구분했는가.
3. 수익 전 임계점과 버텨야 할 과정이 명확한가.
4. 프롬프트를 문장 모음이 아니라 작업 프로토콜로 설명했는가.
5. 수정 난이도와 지원·업데이트시간을 원가에 포함했는가.
6. 시장 자료 미확인 부분을 사실처럼 단정하지 않았는가.
