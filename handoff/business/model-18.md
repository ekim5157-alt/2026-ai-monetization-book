## 핵심 결론
소형 AI SaaS·GPT·웹앱의 구매자는 AI를 사는 것이 아니라 반복되는 한 가지 일을 더 빨리 끝내는 결과를 산다. 현재 운영 시장에서는 영상·PDF·웹페이지 요약, 회의·녹음 정리, AI 음성·영상 제작, 블로그·자기소개서·이미지 생성, 특정 문서 기반 질의응답, 업종별 상담·견적·보고서 작성이 반복적으로 확인된다.

성패는 가입자 수보다 `첫 결과 완료율 → 같은 문제의 반복 사용 → 유료 한도 도달 → 결제 성공 → 다음 달 잔존 → 해지·환불 → 사용자당 AI·서버·지원비`에서 갈린다. 범용 챗봇은 무료 대체재가 많다. 좁은 입력을 받아 사용자가 바로 제출·게시·결정할 수 있는 결과를 만드는 도구가 유료화하기 쉽다.

## 성패를 가르는 것
- 구체적 관찰: LilysAI는 영상·책·PDF·기사·오디오를 넣어 요약과 설명을 받는 한 흐름을 전면에 둔다. Vrew와 Typecast는 음성·자막·영상 파일이라는 명확한 산출물을 제공한다. 뤼튼은 유튜브 요약·실시간 녹음·이미지·블로그·자기소개서처럼 목적별 도구를 분리한다.
- 남들이 놓치는 숨은 변수: 회원가입 전에 가치가 보이는가, 첫 입력 자료를 사용자가 바로 준비할 수 있는가, 결과가 편집·복사·다운로드 가능한가, 같은 사용자가 다음 주에도 같은 일을 하는가.
- 작동 메커니즘: 좁은 문제는 입력 형식·출력 형식·품질 기준을 고정할 수 있어 지원비와 모델 비용을 통제한다. 범용 대화는 요청이 넓어 오류·지원·비용 변동이 커진다.
- 임계점·시간차·교환비용: 무료 사용자가 많아도 유료 한도를 만나는 사용자가 적으면 매출이 없다. 결제 뒤 30~90일 잔존이 확인되기 전에는 광고비를 확대하지 않는다.
- 초보자가 먼저 측정할 것: 방문자 대비 첫 결과 완료, 첫 결과까지 걸린 시간, 7일 안 재사용, 무료 한도 도달, 결제 성공, 30일 잔존, 해지 사유, 사용자당 모델·서버·지원비.
- 사업을 계속할 조건: 유료 사용자가 한 달에 여러 번 같은 핵심 작업을 완료하고, 월 매출총이익이 지원시간과 인프라비를 감당한다.
- 중단·전환 신호: 가입은 많고 첫 결과가 적음, 무료 체험 뒤 재방문이 없음, 한 사용자의 긴 입력 때문에 비용이 급증함, 결과를 매번 사람이 고쳐 줘야 함, 결제 실패·환불·문의가 매출보다 빠르게 늘어남.

### 현재 운영 사례에서 반복되는 패턴
- 시장 반복 패턴: LilysAI는 사용자가 자료를 넣고 즉시 요약 결과를 받는 단일 핵심 행동을 앞에 둔다. 공식 페이지는 120만 명 이상 도달이라는 자체 홍보 수치를 표시하지만 유료 사용자 수와 매출은 공개하지 않는다.
- 시장 반복 패턴: Typecast는 700여 개 음성 캐릭터, 대본 입력, 감정·속도 조절, 자동 자막, 음성·영상 다운로드를 한 제작 흐름으로 묶고 무료 체험→개인 플랜→기업 API로 확장한다.
- 시장 반복 패턴: Vrew는 자막 기반 영상 편집과 AI 생성 기능을 하나의 결과물인 영상 파일로 연결한다.
- 시장 반복 패턴: 뤼튼은 하나의 빈 채팅창보다 유튜브 요약, 실시간 녹음, 이미지 제작, 블로그, 자기소개서처럼 목적별 진입점을 함께 노출한다.
- 포화된 부분: 범용 글쓰기, 일반 요약, 프롬프트만 감싼 챗봇, 차별 없는 이미지 생성.
- 아직 남은 기회: 한국 업무 서식, 특정 업종 데이터, 승인 가능한 출력, 반복 기록, 팀 협업, 기존 프로그램과의 연결이 포함된 좁은 도구.

## 발견·결제·정산 경로
`검색·SNS·콘텐츠·소개 → 무료 첫 결과 → 반복 사용 → 무료 한도·기능 제한 도달 → 카드·앱 결제 → 구독 갱신 → 환불·해지·결제 실패 반영 → PG·앱마켓 수수료 차감 입금`

- 실제 돈을 내는 사람: 시간을 반복적으로 절약하는 개인 사용자, 팀 관리자, 기업 구매·IT 부서.
- 결제수단: 국내 신용·체크카드, 간편결제, 앱스토어 인앱결제, 기업 세금계산서·계좌이체. 제품별 지원 여부는 결제 화면 확인.
- 수익 지급자: 직접 웹 결제는 PG사, 인앱결제는 앱마켓, 기업 계약은 고객사.
- 확정 수익: 무료 체험 종료·구독 결제 성공·환불기간·차지백·세금 처리를 거친 금액.
- 실제 입금: PG·앱마켓·계약의 정산 주기에 따라 계좌 입금. 결제 매출과 통장 입금일이 다르다.

## 수익 전까지 필요한 작업량과 기간
1. 문제 인터뷰 10~20건: 지금 어떻게 처리하고 얼마의 시간이 드는지 확인한다.
2. 수동 서비스 5~10건: 코드보다 먼저 고객이 실제로 원하는 결과 형식을 확인한다.
3. 첫 결과 PoC: 회원가입부터 결과까지 한 흐름만 구현한다.
4. 결제 실험: 무료 한도와 유료 이유를 정하고 실제 결제 5~20건을 확인한다.
5. 30~90일 잔존: 반복 사용과 해지 사유를 확인한다.
6. 비용 상한: 긴 입력·재생성·파일 저장·고객지원의 사용자당 상한을 둔다.

## 사용자당 손익
`사용자당 월 공헌이익 = 결제액 - PG·앱마켓 수수료 - AI 호출비 - 서버·DB·스토리지 배분 - 이메일·분석도구 - 환불·차지백 - 고객지원 시간원가`

- 월 구독 1만원이라도 긴 영상·PDF 처리비와 문의 대응이 1만원을 넘으면 손실이다.
- 사용하지 않는 가입자는 비용이 적지만 잔존 신호도 아니다. 실제 핵심 작업 완료 횟수를 본다.
- 기업 고객은 단가가 높지만 보안 검토·계약·교육·세금계산서·지원 시간이 늘어난다.

## 초보자가 잘못 읽기 쉬운 숫자
- 가입자 수: 첫 결과 완료나 결제 사용자를 뜻하지 않는다.
- 월간 방문자: 반복 핵심 사용자와 다르다.
- 무료 사용량: 유료 의향과 다르며 모델 비용만 늘 수 있다.
- 매출: 환불·PG 수수료·AI비·지원비 차감 전 수치다.
- 총 구독자: 결제 실패·휴면·할인·연간 선결제를 구분하지 못한다.
- 이탈률: 월초·월말 기준과 체험 사용자를 포함하는 방식에 따라 달라진다.

## 반복 가능한 구조
- 입력과 출력이 고정된 한 작업부터 시작한다.
- 첫 결과 전 회원가입·설정 단계를 최소화한다.
- 결과에 출처·근거·편집·다운로드를 제공한다.
- 고비용 요청은 길이·파일 수·해상도·재생성 횟수로 제한한다.
- 결제 실패 알림, 카드 갱신, 해지 이유, 데이터 내보내기·삭제를 제품 기능으로 둔다.
- 사람이 개입하는 지원 요청을 유형화하고 제품 개선과 유료 서비스로 나눈다.

## 공식 확인 사실
- LilysAI 공식 페이지는 영상·책·PDF·기사·오디오를 넣어 요약·설명·심화 결과를 제공하는 현재 서비스를 운영한다.
- Typecast 공식 페이지는 대본 입력, 700여 개 캐릭터, 감정·속도 조절, 자동 자막, 음성·영상 다운로드, 기업 API를 안내한다.
- 뤼튼 현재 화면은 채팅 외에 유튜브 요약·실시간 녹음·이미지 제작·블로그·자기소개서 등 목적별 도구를 노출한다.
- 제품의 자체 홍보 사용자 수·비용 절감 표현은 개별 운영자의 유료 전환·순이익 근거로 사용하지 않는다.

## 조사 결과 때문에 수정한 기존 가정
- `작은 기능을 만들면 구독자가 붙는다`는 가정을 폐기한다. 반복 문제와 유료 한도 도달을 먼저 확인한다.
- 범용 챗봇보다 첫 결과가 명확한 목적별 흐름을 우선한다.
- 가입자·다운로드 수보다 핵심 작업 완료·7일 재사용·30일 잔존을 우선한다.
- 서버비만 원가로 보지 않고 AI 사용량·파일 처리·지원·환불·결제 실패를 포함한다.
