## 핵심 결론
소형 AI SaaS·GPT·웹앱은 기능 수가 아니라 **반복 문제 × 첫 가치 도달 × 사용자당 결제액과 AI·서버 비용의 차이 × 잔존·해지·결제 실패 × 지원·보안 비용**의 사업입니다.

`활성 유료 사용자 × 사용자당 결제액 - AI·서버·PG·환불·지원·광고·개발비 = 월 순이익`

데모가 잘 작동해도 사용자가 다시 오지 않거나 사용자당 비용이 결제액보다 크면 사업이 아닙니다.

## 성패를 가르는 것

### 좁고 반복되는 문제
- 관찰: 범용 챗봇은 기존 도구와 차이가 약합니다.
- 숨은 변수: 주·월 사용 빈도, 오류비용, 입력자료, 결과의 다음 사용 단계입니다.
- 작동 메커니즘: 한 직무의 반복 문제를 구조화된 결과로 끝내면 재방문이 생깁니다.
- 실행 판단: `누가 어떤 자료를 넣고 어떤 결과를 몇 번 다시 쓰는가`를 한 문장으로 정합니다.
- 실패·전환 신호: 첫 주 뒤 재사용이 없으면 기능보다 문제를 바꿉니다.

### 첫 가치 도달
- 관찰: 가입과 무료체험은 결과가 아닙니다.
- 숨은 변수: 가입 단계, 초기 입력량, 첫 결과시간, 예시 데이터, 오류 복구입니다.
- 작동 메커니즘: 가입 직후 실제 업무 결과를 만들면 유료 전환 가능성이 높아집니다.
- 실행 판단: 10분 안에 첫 결과가 나오게 하고 가입→첫 입력→결과→저장→재사용을 측정합니다.
- 실패·전환 신호: 가입은 늘지만 첫 결과 완료율이 낮으면 온보딩을 줄입니다.

### 사용자당 비용 상한
- 관찰: 긴 문서·이미지·음성·반복 대화는 AI 호출비와 저장비를 키웁니다.
- 숨은 변수: 입력·출력 길이, 모델, 재시도, 캐시, 무료 한도, 악용입니다.
- 작동 메커니즘: 기능별 평균·최대 비용을 측정해야 가격과 사용량 제한을 설계할 수 있습니다.
- 실행 판단: `호출비 + 재시도 + 저장비 + 지원시간`을 사용자별로 기록합니다.
- 실패·전환 신호: 활성 사용자가 늘수록 사용자당 손실이 커지면 가격·모델·기능을 바꿉니다.

### 잔존·해지·결제 실패
- 관찰: 첫 결제는 반복 가치가 아닙니다.
- 숨은 변수: 월 사용 빈도, 해지 이유, 카드 실패, 재시도, 환불, 재활성입니다.
- 작동 메커니즘: 결과를 저장·재사용할수록 잔존이 높아집니다.
- 실행 판단: 신규·첫 결과·활성·유료·잔존·해지·결제 실패를 분리합니다.
- 실패·전환 신호: 신규 결제와 해지가 같은 속도로 늘면 광고를 중단합니다.

### 지원·장애·보안
- 관찰: 로그인·결제·파일·외부 API 오류가 문의를 만듭니다.
- 숨은 변수: 로그, 응답시간, 장애 공지, 환불 기준, 데이터 보관·삭제, 관리자 권한입니다.
- 작동 메커니즘: 로그·도움말·삭제 절차가 없으면 운영비와 신뢰 위험이 커집니다.
- 실행 판단: 오류 코드, 상태 안내, FAQ, 역할별 권한, 백업·삭제·사고 대응을 준비합니다.
- 실패·전환 신호: 지원시간이 사용자당 결제액을 넘거나 데이터 위치를 설명하지 못하면 기능을 줄입니다.

### 제품 시연과 원소스멀티유즈
- 관찰: 화려한 화면보다 반복 업무의 전후 결과가 구매를 만듭니다.
- 숨은 변수: 예시 데이터, 전후 수치 근거, 채널별 첫 장면, 링크 마찰, 유입별 잔존입니다.
- 작동 메커니즘: 한 사례를 채널별 목적에 맞게 바꾸면 유입비용을 분산할 수 있습니다.
- 실행 판단: 유튜브는 전체 사용법, 인스타그램은 전후 화면, 틱톡은 빠른 결과, X는 핵심 수치, 페이스북은 업종 사례로 재가공합니다.
- 실패·전환 신호: 조회만 늘고 첫 결과·유료 전환·잔존이 늘지 않으면 채널을 줄입니다.

## 핵심 현실 인사이트
- 구체적 관찰: 가입·첫 결제·활성·잔존·순이익은 서로 다른 사건입니다.
- 남들이 놓치는 숨은 변수: 첫 결과시간, 사용자당 최대 비용, 결제 실패, 지원시간, 데이터 삭제입니다.
- 결과가 달라지는 작동 메커니즘: 반복 문제를 좁히고 비용·오류를 통제하면 사용자가 늘어도 마진이 남습니다.
- 임계점·시간차·교환비용: `콘텐츠 → 가입 → 첫 결과 → 결제 → 반복 사용 → 해지·환불·결제 실패 → PG 정산 → 원화 계좌 입금`
- 초보자가 먼저 측정하거나 결정할 것: 문제 빈도, 첫 결과, 비용 상한, 결제 방식, 지원 범위입니다.
- 계속할 조건: 활성·잔존이 개선되고 비용 차감 이익이 남는 경우입니다.
- 중단·전환 신호: 가입만 늘고 재사용이 없거나 사용자가 늘수록 손실이 커지는 경우입니다.
- 대표 사례·수치·반례: 월 1회 범용 챗봇보다 매주 쓰는 견적·보고·분류 도구가 더 높은 결제 가치를 만들 수 있습니다.
- 반복 확인된 구조인지 단일 사례의 편집 판단인지: 잔존·사용자당 비용을 우선하는 것은 사업성 판단이며 PG·API 조건은 공식 확인이 필요합니다.

## 시장 현실
- 네이버 블로그·카페·브런치·유튜브·커뮤니티에서 먼저 확인한 동향: 범용 AI보다 한 업무를 끝내는 작은 도구가 판매가 쉽습니다.
- 반복된 패턴: 기능·가입자를 강조하고 잔존·AI 비용·지원비를 공개하지 않는 사례가 많습니다.
- 상충되는 의견: 월 구독과 건별 결제 중 유리한 방식은 사용 빈도에 따라 달라집니다.
- 작성자 이해관계 또는 광고 가능성: 개발 도구·호스팅 판매자는 출시 속도를 강조할 유인이 있습니다.
- 이미 포화된 부분: 범용 챗봇과 저장·업무연결이 없는 일회성 생성기입니다.
- 아직 남은 기회: 한글 업무·원화 결제·저장·재사용·세팅 지원을 결합한 좁은 도구입니다.

## 판매·수익·광고 경로
- 실제 구매자·독자·시청자·광고주: 소상공인, 1인 사업자, 프리랜서, 교육·상담·쇼핑몰·기관 실무자
- 발견·유입 경로: 네이버 검색·블로그, 다섯 영상 채널, 당근, 크몽, 위시켓, 기존 고객
- 판매 또는 광고가 일어나는 플랫폼: 자체 웹앱·판매페이지, 크몽·위시켓, 당근 비즈니스 프로필·지역광고
- 결제수단: 토스페이먼츠·KG이니시스·나이스페이의 신용카드·간편결제·계좌 관련 수단
- 수익 지급자: PG사 또는 판매 플랫폼
- 플랫폼 수수료·광고비·환불·CS 비용: AI API·서버·DB·저장·PG·환불·광고·지원·개발·보안
- 확정 수익 조건: 환불·결제 실패·해지 조건을 통과하고 결제액이 확정된 경우
- 실제 입금 시점과 경로: PG·플랫폼 정산 뒤 원화 계좌 입금
- 반복 가능한 구조인지: 사용자당 마진과 잔존이 안정되어야 합니다.
- 대상 범위 교정: 한글 사용자·원화 결제·원화 정산을 기준으로 합니다.

## 공식 확인 사실
PG 가입·정기결제·수수료·정산, AI API·서버·데이터 보관·개인정보·전자상거래 조건은 다음 단계가 공식 문서로 확인합니다.

## 현실 기준 교정
- 미국 기준 자료인지: 대응되는 구매자·원화 결제·정산이 없는 외부 사례는 실행 모델에서 제외합니다.
- 판매자 형태별 적용 가능 여부: 개인과 개인사업자의 PG 심사·통신판매업·세금 조건을 확인합니다.
- 결제·지급·정산 조건: 일반·정기결제·실패·재시도·환불·부분취소를 구분합니다.
- 원화 가격·세금·수수료: PG·AI·서버·광고·부가가치세·종합소득세를 포함합니다.
- 언어 지원·지역 제한: 한글 입력·출력·고객지원을 기준으로 합니다.
- 플랫폼·소비 습관 차이: 네이버·당근·크몽·위시켓 유입의 결제 의도가 같다고 가정하지 않습니다.
- 적용 확인이 필요한 항목: 정기결제, 개인정보, 외부 API 전송, 환불·장애 기준입니다.
