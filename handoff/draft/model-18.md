# 모델 18. 소형 AI SaaS·GPT·웹앱

- 상태: 초보자 실행 원고 v2
- 선행 문서: `handoff/business/model-18.md`, `handoff/research/model-18.md`
- 문장 형식 기준: `final/model-01.md`
- 공식 확인일: 2026-06-22

## 1. 한 줄 결론

소형 AI SaaS·GPT·웹앱은 **반복되는 좁은 문제 × 10분 안의 첫 가치 × 사용자당 AI·서버·결제·지원 원가 × 7일·30일 재사용 × 해지·결제 실패·데이터 삭제 대응**이 맞아야 수익모델이 됩니다.

## 2. 이 모델은 무엇인가

영상·PDF·웹 요약, 회의 정리, AI 음성·영상 제작, 목적별 문서 작성, 특정 문서 기반 질의응답, 업종별 견적·보고처럼 반복 업무 하나를 빠르게 끝내 주는 소형 도구입니다. 가입자 수보다 첫 결과·재사용·유료 전환·잔존이 중요합니다.

## 3. 성패를 가르는 것

`누가 어떤 자료를 넣고 어떤 결과를 몇 번 다시 쓰는가`를 한 문장으로 정의합니다. 회원가입 전후 입력을 줄이고 예시 데이터로 첫 결과를 빠르게 보여 줍니다.

기능별 평균·최대 API·저장·전송·지원비를 측정하고 무료 한도와 긴 입력·파일·재생성 상한을 둡니다. 결제 요청·인증·승인을 서버에서 검증하고 결제 상태와 서비스 권한을 대조합니다.

## 4. 누가 돈을 지급하는가

개인 사용자, 팀 관리자와 기업 고객이 PG·앱마켓·계좌이체로 결제하고 환불·부분취소·결제 실패·해지 조정 뒤 정산됩니다.

## 5. 실행 경로

문제 인터뷰 10~20건과 수동 서비스 5~10건으로 원하는 결과 형식을 확인합니다. 한 입력→한 결과 흐름의 PoC를 만들고 사용자 5~10명에게 첫 결과·재사용·비용을 관찰합니다.

첫 결제 5~20건과 30~90일 잔존을 확인하기 전 기능과 광고를 늘리지 않습니다.

## 6. 국내 운영 기준

한글 사용자, 원화 결제와 원화 정산을 기준으로 합니다. 국내 카드·계좌·세금계산서와 개인정보 보유·삭제 절차를 우선 설계합니다.

## 7. 주요 타겟

같은 견적·보고·분류·상담 준비를 반복하는 1인 사업자·실무자·팀에게 적합합니다.

## 8. 부가 타겟

기존 구축 고객의 공통 기능을 표준 제품으로 바꾸거나 초기 세팅 서비스를 함께 판매하려는 개발자·컨설턴트에게 적합합니다.

## 9. 수익구조

월 구독, 건별 크레딧, 팀 요금제, 초기 세팅과 맞춤 연동으로 나눕니다. AI·서버·DB·저장·PG·환불·광고·지원비를 차감합니다.

## 10. 첫 수익 예상

기존 고객 문제가 분명하면 1~8주 안에 초기 세팅이나 첫 유료 사용자를 만들 수 있습니다. 반복 사업성은 30일 잔존과 사용자당 공헌이익이 확인된 뒤 판단합니다.

## 11. 실제 입금 예상

`유입 → 첫 결과 → 반복 사용 → 결제 → 해지·환불·결제 실패 → PG 정산 → 입금` 순서로 기록합니다.

## 12. 필요한 제품과 공식 링크

OpenAI API — ChatGPT 구독과 별도 사용량 과금. 모델·검색·파일·데이터 보존 조건과 사용자별 비용 상한 확인. https://platform.openai.com/

CLOVA Studio — 한글 업무용 모델 후보. 원화 단가는 최신 계정 화면 확인. https://www.ncloud.com/product/aiService/clovaStudio

토스페이먼츠 — 일반·자동결제·취소 API 제공. 자동결제는 추가 계약과 직접 스케줄링 필요. https://www.tosspayments.com/

NAVER Cloud Object Storage — 저장량·요청·전송 종량 과금. https://www.ncloud.com/

## 13. 오늘 실행

반복 문제 하나, 첫 결과, 사용자당 비용 상한과 결제 방식을 정합니다.

## 14. 3일 실행

최소 화면과 기능별 비용·오류·지원 기록표를 만듭니다.

## 15. 7일 실행

사용자 5~10명의 첫 결과 완료·재사용·최대 비용을 관찰합니다.

## 16. 30일 실행

약 20명의 활성·유료·잔존·해지·결제 실패·지원시간·순이익을 비교합니다.

## 17. 자동화

온보딩, 사용량 집계, 결제 상태 대조, 실패 알림, 탈퇴·삭제 요청 흐름을 자동화할 수 있습니다. 고비용 기능에는 상한을 둡니다.

## 18. 사람이 직접 할 일

사람은 문제 선택, 첫 가치, 가격, 지원 범위, 데이터 보유·삭제, 관리자 접근과 장애·환불 기준을 결정합니다.

## 19. 실패 원인

범용 챗봇을 만드는 것, 가입자를 가치로 보는 것, 무료 사용량과 최대 비용을 측정하지 않는 것, 결제 상태와 서비스 권한이 불일치하는 것이 대표적인 실패입니다.

## 20. 중단·전환 기준

첫 결과 뒤 재사용이 없거나 사용자가 늘수록 손실이 커지고 지원시간이 결제액을 넘으면 문제·기능·가격을 바꿉니다.

## 21. 위험과 주의사항

API 학습 제외와 무보존을 혼동하지 않습니다. 결제 인증만으로 서비스를 제공하지 않고 승인과 금액을 서버에서 확인합니다. 해지·환불·데이터 삭제 절차를 제공합니다.
