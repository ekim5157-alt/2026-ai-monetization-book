# 모델 22. AI 대량등록 구매대행·오픈마켓 자동화

- 상태: 초보자 실행 원고 v2
- 선행 문서: `handoff/business/model-22.md`, `handoff/research/model-22.md`
- 문장 형식 기준: `final/model-01.md`
- 공식 확인일: 2026-06-22

## 1. 한 줄 결론

AI 대량등록 구매대행·오픈마켓 자동화는 **활성 상품 비율 × 주문 발생 SKU 비율 × 공급가·재고 동기화 × 배송·반품 정확성 × 천 개 활성 등록당 확정 순이익 × 자동화 중단·복구 능력**이 맞아야 수익모델이 됩니다.

## 2. 이 모델은 무엇인가

국내외 공급처 상품 데이터를 쿠팡·스마트스토어·오픈마켓에 대량등록하고, 등록·갱신·오류 탐지·확대·중단을 자동화하는 운영 모델입니다. 총 등록 수보다 실제 판매 가능한 활성 상품과 천 개당 순이익이 중요합니다.

## 3. 성패를 가르는 것

총 등록→활성 등록→노출→주문 발생→배송→구매확정→비용 차감 순이익을 단계별로 기록합니다. 상품별 공급처·최근 확인일·공급가·재고·옵션코드·배송비·판매중지 기준을 관리합니다.

SKU마다 최저 허용가와 공급가·환율·배송비 급등 시 중지 기준을 둡니다. `공급처 원본 → AI 정리 → 규칙 검사 → 사람 승인 → 게시 → 오류 기록` 순서를 사용합니다.

## 4. 누가 돈을 지급하는가

국내 구매자가 오픈마켓에서 결제하고 각 플랫폼 정산 시스템이 판매자에게 지급합니다. 공급처·구매대행사·배송대행사에는 판매자가 먼저 지급할 수 있습니다.

## 5. 실행 경로

상품군 하나, 공급처 1~2곳, 채널 1~2개로 시작합니다. 100개 안팎을 시험 등록해 활성 비율·오류·노출·주문·품절·배송을 확인합니다.

주문이 발생한 상품만 상세페이지·광고·공급처를 추가 검토하는 2단계 구조로 운영합니다. 주문·송장·배송·취소·반품·정산을 하나의 상태표로 연결합니다.

## 7. 주요 타겟

공급처 데이터와 주문 후 구매·배송·CS를 직접 관리할 수 있고 선결제 자금을 감당할 수 있는 판매자에게 적합합니다.

## 8. 부가 타겟

규격이 세분된 롱테일·전문상품과 안정된 공급 데이터를 가진 상품군 운영자에게 적합합니다.

## 9. 수익구조

판매가와 공급·배송·판매비용의 차이가 수익입니다. 공급가·구매대행비·배송비·판매비용·광고·반품·불량·CS·도구·세금을 차감합니다.

## 10. 첫 수익 예상

첫 주문은 1~4주 안에 가능할 수 있으나 확정수익은 배송·구매확정·반품 뒤에 결정됩니다. 확대는 30일 이상 운영지표를 확인한 뒤 진행합니다.

## 11. 실제 입금 예상

`데이터 수급 → 등록 → 주문 → 공급처 선결제 → 배송 → 구매확정·반품 → 플랫폼 정산 → 입금` 순서로 기록합니다.

## 12. 필요한 제품과 공식 링크

쿠팡 OPEN API — 상품·주문·배송·반품·CS·정산 연동. 필수정보와 호출 제한 확인. https://developers.coupangcorp.com/

도매매·스피드고전송기 — 상품 DB·전송·주문·품절·가격변동 관리. https://domeme.com/ · https://speedgo.domeggook.com/

ESM PLUS — G마켓·옥션 상품·주문·정산 관리. https://www.esmplus.com/

Google Sheets — 활성 등록·천 개당 주문·순이익·현금흐름 기록. https://sheets.google.com/

## 13. 실행계획

### 오늘 실행

상품군 하나, 공급처 1~2곳, 채널 1개, 최저 허용가와 운전자금 상한을 정합니다.

### 3일 실행

원본 데이터·옵션코드·가격·재고 갱신 흐름과 판매중지 기준을 만듭니다.

### 7일 실행

100개 안팎을 시험 등록해 오류·활성 비율·주문·공급처 취소를 기록합니다.

### 30일 실행

활성 등록·천 개당 주문·순이익·품절·배송·반품·현금 묶임을 비교합니다.

## 14. 자동화

상품 정리·전송·가격·재고 갱신·주문 수집·상태 기록을 자동화할 수 있습니다. 재시도·중복 방지·누락 대조·알림을 둡니다.

## 15. 사람이 직접 할 일

사람은 공급처 사용 범위, 규격·인증·상표·이미지, 최저가격, 주문 결제, 배송·반품과 현금흐름을 검수합니다.

## 16. 실패 원인

총 등록 수만 늘리는 것, 공급가·재고 갱신이 늦는 것, 손실 가격을 자동 적용하는 것, 주문 이후 흐름과 선결제 자금을 빼는 것이 대표적인 실패입니다.

## 17. 중단·전환 기준

활성 등록이 줄거나 주문 뒤 품절·가격변경 취소가 많고 자동화 장애를 감지하지 못하면 등록 확대를 멈춥니다.

## 18. 위험과 주의사항

공급처의 데이터·이미지 사용기간과 공급 종료 뒤 삭제 조건을 확인합니다. API 사양·호출 제한 변경과 주문 재수집에 대비합니다.
